# 1．引言

## 1.1 编写目的

本文档描述 CompeteHub 的软件需求，包括产品目标、用户特征、运行环境、数据对象、功能需求、非功能需求和约束条件。本文档用于指导设计、开发、测试和验收。

## 1.2 项目背景

本科学生获取竞赛信息时面临来源分散、筛选成本高、适配判断困难和关键时间节点容易遗漏等问题。CompeteHub 通过结构化赛事信息、搜索筛选、规则推荐、收藏订阅和站内提醒，帮助学生高效发现适合自己的竞赛。

管理员作为辅助角色，负责赛事录入、审核发布、状态维护、基础字典配置和操作记录管理。

## 1.3 定义

| 术语 | 定义 |
|---|---|
| 学生端 | 面向本科学生的搜索、筛选、推荐、订阅和提醒界面。 |
| 管理端 | 面向管理员的赛事录入、审核、状态管理和配置界面。 |
| 已发布赛事 | 已通过审核并允许在学生端展示的赛事。 |
| 规则推荐 | 基于学生画像、赛事标签、时间紧迫度和热度等显式规则生成的推荐。 |
| 推荐理由 | 系统向用户说明推荐依据的文本，例如“与专业匹配”“近期截止”。 |

# 2．总体概述

## 2.1 软件介绍

CompeteHub 是一个大学生竞赛信息智能筛选与推荐系统。系统以学生查赛、判断适配、订阅提醒为主线，以管理员审核发布和信息维护为支撑，形成可信、可筛选、可跟进的赛事信息服务。

## 2.2 软件功能概述

系统核心功能包括：

- 用户注册登录与学生画像维护。
- 管理员赛事人工录入、审核发布和状态管理。
- 赛事关键词搜索、多维筛选、排序和分页。
- 赛事详情展示、来源链接、时间节点、适配标签和官方通道。
- 收藏订阅、个人赛事日历和站内提醒。
- 基于规则的个性化推荐和推荐理由展示。
- 审核记录、操作日志和基础配置管理。

## 2.3 用户特征

| 用户 | 特征 | 主要目标 |
|---|---|---|
| 本科学生 | 对竞赛信息有需求，但时间和筛选能力有限 | 快速发现适合赛事并避免错过节点 |
| 管理员 | 熟悉赛事来源和学校发布流程 | 维护可信、完整、状态准确的赛事信息 |
| 教师或教学秘书 | 关注赛事覆盖和学生参与情况 | 后续查看热度和推荐效果 |
| 竞赛组织者或社团 | 关注赛事宣传和答疑 | 后续沉淀资料和经验 |

## 2.4 运行环境

客户端运行在现代浏览器中。服务端包括 Flask API、PostgreSQL、Redis 和 Celery worker。本地开发通过 `just`、`uv`、npm 和 Docker Compose 管理。

# 3．数据描述

## 3.1 数据建模

核心实体包括：

- `User`：账号身份、角色和状态。
- `StudentProfile`：学生学院、专业、年级、兴趣标签、竞赛经历和目标偏好。
- `Competition`：赛事标题、来源、主办方、类别、条件、详情、状态和创建人。
- `CompetitionTimeNode`：报名开始、报名截止、提交截止、比赛开始、评审答辩和结果公布等节点。
- `CompetitionTag` 与 `CompetitionTagLink`：赛事分类、适配、级别和认可度等标签。
- `Favorite`：学生收藏关系。
- `Subscription`：学生订阅关系和提醒偏好。
- `Reminder` 与 `Message`：提醒计划和站内消息。
- `ReviewRecord` 与 `AuditLog`：审核过程和操作日志。
- `RecommendationRule`：推荐规则、权重和理由模板。

## 3.2 数据字典

| 数据项 | 类型 | 说明 |
|---|---|---|
| `role` | 枚举 | `student`、`admin`，预留 `teacher`、`organizer`。 |
| `competition.status` | 枚举 | 草稿、待审核、已发布、已驳回、已下架、已归档、已取消、已过期。 |
| `participant_form` | 枚举或字典 | 个人赛、团队赛或不限。 |
| `node_type` | 枚举 | 报名开始、报名截止、作品提交、比赛开始、评审答辩、结果公布。 |
| `reminder.status` | 枚举 | 待发送、已发送、已读、已取消、发送失败。 |
| `tag_type` | 枚举 | 级别、适配、类别、认可度、形式等。 |

# 4．功能需求

## 4.1 功能需求概要

| 编号 | 功能 | 优先级 |
|---|---|---|
| FR1 | 用户注册登录与画像维护 | 核心 |
| FR2 | 管理员赛事录入、审核和状态管理 | 核心 |
| FR3 | 赛事搜索、筛选、排序和分页 | 核心 |
| FR4 | 赛事详情、适配标签和官方通道 | 核心 |
| FR5 | 收藏订阅、个人日历和站内提醒 | 核心 |
| FR6 | 规则推荐和推荐理由 | 核心 |
| FR7 | 审核记录、操作日志和基础配置 | 增强 |
| FR8 | 资料归档、常见问题和往届经验 | 增强 |
| FR9 | 自动采集、多渠道通知和模型推荐 | 后续 |

## 4.2 功能需求详细说明

### FR1 用户注册登录与画像维护

学生可使用学号、邮箱或手机号注册登录。登录后可维护学院、专业、年级、兴趣方向、竞赛经历和目标偏好。系统应保证学号、邮箱和手机号唯一。未登录用户访问收藏、订阅和画像编辑时应被引导登录。

### FR2 管理员赛事录入、审核和状态管理

管理员可从可信来源人工录入赛事，填写标题、来源、主办方、类别、时间节点、报名条件、官方链接、附件和适配信息。赛事可保存为草稿或提交审核。审核通过后进入已发布状态并在学生端可见。驳回、下架、归档、取消和过期操作必须记录原因和操作日志。

### FR3 赛事搜索、筛选、排序和分页

学生可按赛事名称、简称、主办方、类别和正文摘要进行关键词搜索。系统支持按专业、年级、赛事类别、标签、报名状态、截止日期和参赛形式筛选。排序方式包括截止时间、发布时间、推荐度和热度。结果必须只展示当前用户可见的赛事。

### FR4 赛事详情、适配标签和官方通道

赛事详情页展示来源、来源链接、官方报名链接、主办方、类别、适合专业、适合年级、参赛形式、团队人数、报名条件、材料要求、关键时间节点、价值说明和注意事项。官方链接跳转应清晰标识外部来源。

### FR5 收藏订阅、个人日历和站内提醒

学生可收藏赛事，也可订阅赛事关键节点。收藏和订阅是独立概念。订阅后系统基于赛事时间节点和用户提醒配置生成待发送提醒，并在个人赛事日历和消息中心展示。取消订阅应取消未来未发送提醒。

### FR6 规则推荐和推荐理由

系统根据学生画像、赛事标签、适合专业、适合年级、兴趣匹配、截止时间和热度生成推荐列表。每条推荐应给出可解释理由。画像不足时系统可降级为热门或近期截止赛事。

### FR7 审核记录、操作日志和基础配置

系统应记录赛事创建、修改、提交审核、通过、驳回、下架、归档、取消等操作。管理员可维护赛事类别、标签、专业、年级、提醒节点类型等基础字典。

## 4.3 功能需求分级

核心能力必须优先交付，以保证学生查赛、判断、订阅、提醒和管理员发布的主流程闭环。增强能力根据开发进度逐步实现。后续能力作为系统演进方向，不作为 v0.0 验收前提。

# 5．非功能需求

| 类别 | 要求 |
|---|---|
| 可用性 | 学生应在 3 次以内筛选操作中定位符合条件的赛事。 |
| 可靠性 | 提醒数据以 PostgreSQL 为事实来源，避免仅依赖 Redis。 |
| 安全性 | 后端必须执行角色权限校验；前端权限判断仅用于体验。 |
| 可维护性 | 后端按 routes、schemas、services、repositories、models 分层。 |
| 可扩展性 | 推荐、多渠道通知和采集能力应通过明确扩展点逐步加入。 |
| 可测试性 | 后端提供单元测试和接口测试；前端提供 lint 和 build 校验。 |
| 可追溯性 | 赛事审核、状态变更和配置变更必须写入日志。 |

# 6．约束条件

## 6.1 业务约束

- 只有已发布赛事默认出现在学生端搜索、列表和推荐中。
- 来源名称和来源链接是赛事发布的必填信息。
- 推荐理由必须能追溯到规则或赛事字段。

## 6.2 硬件约束

v0.0 可在普通开发电脑运行。部署演示环境时需要能够运行 Web、API、PostgreSQL 和 Redis 的服务器或容器环境。

## 6.3 软件约束

- 前端使用 Vue 3、Vite、TypeScript、Vue Router 和 Pinia。
- 后端使用 Flask、SQLAlchemy、Celery 和 Redis。
- 数据库使用 PostgreSQL。
- 本地命令优先使用 `just` 和 `scripts/agent-env.sh`。

## 6.4 其他约束

- 文档和实现发生实质变化时应同步更新。
- 新增 CI workflow 时需要更新 `.github/workflows/README.md`。
- 报告文档属于课程材料，不覆盖 `docs/` 中的稳定产品和技术文档。
