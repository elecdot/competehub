# 1．引言

## 1.1 编写目的

本文档记录 CompeteHub 在 v0.0 阶段的开发过程、阶段成果、工时估计、进度对比、技术评价和经验教训，为课程检查、复盘和后续迭代提供依据。

## 1.2 项目背景

项目目标是建设一个大学生竞赛信息智能筛选与推荐系统，优先解决学生查找可信赛事、判断适配性、订阅关键节点和接收站内提醒的问题。管理员负责赛事录入、审核发布和状态维护。

# 3．开发结果

## 3.1 产品

v0.0 阶段形成的主要产物：

| 类型 | 内容 |
|---|---|
| 前端程序 | `apps/web`，Vue 3/Vite/TypeScript 应用，包含首页、赛事列表、详情、推荐、日历和管理端入口页面。 |
| 后端程序 | `apps/api`，Flask 应用，包含配置、扩展、健康检查、错误处理、核心模型目录和 Celery 任务目录。 |
| 基础设施 | `infra/docker-compose.yml`，提供本地 PostgreSQL 和 Redis。 |
| 文档 | PRD、架构说明、技术规格、API 规格、数据模型、设置说明、工具说明和课程报告草稿。 |
| 自动化 | `justfile`、GitHub Actions、文档构建和前后端验证命令。 |

当前实现重点仍在基础工程和设计文档，核心业务 API 与页面交互需要后续迭代实现。

## 3.2 主要功能及性能

已完成或已具备基础：

- 仓库结构和目录 README。
- 后端 Flask application factory。
- 环境配置与测试配置。
- 数据模型文件组织。
- 健康检查接口和基础测试。
- Celery 任务目录和提醒任务扩展点。
- 前端路由、页面入口、API client 和 store 骨架。
- Docker Compose 本地基础设施配置。
- MkDocs 文档站点构建配置。
- GitHub Actions 文档和 CI 配置。

待完成核心功能：

- 用户注册登录与画像维护。
- 赛事录入、审核、发布和状态管理。
- 搜索筛选、赛事详情、收藏订阅和提醒。
- 规则推荐和推荐理由。
- 管理端表单、审核页面和日志查询。

性能目标：

- 学生在 3 次以内筛选操作中定位符合条件的赛事。
- 收藏或订阅操作控制在 2 步以内。
- 列表接口支持分页、筛选和排序，避免一次返回过大数据集。
- 提醒生成和发送通过异步任务处理，避免阻塞用户请求。

# 4．评价

## 4.1 生产率评价

v0.0 阶段重点放在项目边界和基础工程建设，产出集中在文档、架构、仓库结构和脚手架。该阶段为后续编码降低了沟通成本，但业务功能可演示程度仍有限。

提升生产率的因素：

- monorepo 结构统一，前后端和文档在一个仓库中管理。
- `just` 命令统一了常用验证入口。
- PRD、API 规格和数据模型为后续开发提供了明确依据。

限制生产率的因素：

- 核心业务模块较多，需要控制迭代范围。
- 前后端联调依赖 API 合同稳定。
- 推荐和提醒涉及跨模块数据一致性，需要额外测试。

## 4.2 技术方案评价

技术方案整体适合课程项目和后续扩展：

- Vue 3 + Flask 易于快速开发和演示。
- PostgreSQL 适合保存结构化赛事、用户、订阅、提醒和日志数据。
- Redis + Celery 能支撑提醒发送和过期标记等异步任务。
- 规则推荐优先于机器学习模型，符合当前数据量和可解释性要求。

需要注意的问题：

- 后端权限必须由 API 兜底，不能依赖前端隐藏入口。
- Redis 不能作为提醒、订阅或赛事状态的唯一事实来源。
- 自动采集、多渠道通知和模型推荐需要在核心链路稳定后再引入。

# 5．经验与教训

- 课程项目应优先保证主流程闭环，避免一开始实现论坛、评论、模型推荐等扩展功能。
- 稳定产品文档和课程报告应分离，报告可以引用 PRD 和技术规格，但不应替代它们。
- 数据模型和 API 规格需要尽早确定，否则前后端联调容易重复返工。
- 赛事信息可信度是系统价值基础，来源字段、审核记录和状态流转必须作为核心能力设计。
- 提醒任务涉及时间和幂等性，应尽早编写测试用例覆盖取消订阅、赛事下架和重复执行等情况。
