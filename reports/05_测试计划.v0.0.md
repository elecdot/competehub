# 1．引言

## 1.1 编写目的

本文档说明 CompeteHub 的测试目标、测试范围、测试条件、测试方案、测试用例设计、执行记录方式和质量评价标准。v0.0 作为测试计划初稿，重点覆盖当前基础工程和后续核心业务链路。

## 1.2 项目背景

CompeteHub 面向学生提供赛事搜索筛选、详情查看、规则推荐、收藏订阅、个人日历和站内提醒，并为管理员提供赛事录入、审核发布和状态管理。系统包含前端、后端、数据库、Redis 和异步任务，测试需要覆盖接口、业务规则、权限和用户流程。

## 1.3 定义

| 术语 | 定义 |
|---|---|
| 单元测试 | 针对函数、服务或模型规则的独立测试。 |
| 接口测试 | 针对 REST API 请求、响应、权限和错误码的测试。 |
| 构建测试 | 对前端 build、lint 和文档构建的验证。 |
| 联调测试 | 前后端结合真实接口完成用户流程验证。 |
| 回归测试 | 修复缺陷或新增功能后确认既有能力未被破坏。 |

# 2．测试计划

## 2.1 测试条件

### 2.1.1 软件环境

- Python 3.11。
- Node.js 与 npm。
- `uv` 与 `just`。
- Docker Compose。
- PostgreSQL。
- Redis。
- 现代浏览器。

### 2.1.2 本地启动和验证命令

| 目标 | 命令 |
|---|---|
| 后端依赖同步 | `just api-sync` |
| 后端测试 | `just api-test` |
| 后端 lint | `just api-lint` |
| 前端依赖安装 | `npm --prefix apps/web install` |
| 前端 lint | `just web-lint` |
| 前端构建 | `just web-build` |
| 文档构建 | `just docs-build` |
| 本地基础设施 | `just infra-up` |
| Docker Compose 配置检查 | `docker compose -f infra/docker-compose.yml config` |

### 2.1.3 测试数据

计划准备以下基础测试数据：

- 学生用户：包含不同学院、专业、年级和兴趣标签。
- 管理员用户：具备赛事录入和审核权限。
- 赛事数据：覆盖待审核、已发布、已驳回、已下架、已归档、已取消、已过期等状态。
- 时间节点：覆盖报名开始、报名截止、作品提交截止、比赛开始、评审答辩和结果公布。
- 标签数据：覆盖类别、适配、级别、认可度和参赛形式。

## 2.2 测试方案设计

### 2.2.1 测试范围

v0.0 阶段：

- 仓库命令、文档构建、后端健康检查、前端构建和基础路由。

核心业务阶段：

- 用户注册登录和画像维护。
- 管理员赛事录入、审核发布和状态变更。
- 学生端赛事搜索、筛选、排序、详情查看。
- 收藏、订阅、提醒生成、消息读取和个人日历。
- 规则推荐和推荐理由。
- 权限、错误处理和审计日志。

不在 v0.0 强制测试范围：

- 机器学习推荐。
- 自动采集。
- 邮件、短信、微信等多渠道通知。
- 高并发压测。

### 2.2.2 测试方法

- 静态检查：运行后端 lint、前端 lint 和文档构建。
- 单元测试：覆盖后端服务层、模型状态、推荐规则和提醒生成逻辑。
- 接口测试：使用 Flask test client 验证 API envelope、错误码和权限。
- 前端构建测试：保证 TypeScript、路由和页面构建通过。
- 联调测试：通过浏览器验证学生端和管理端主流程。
- 回归测试：缺陷修复后重新执行相关单元、接口和前端检查。

### 2.2.3 核心测试用例

| 编号 | 测试项 | 输入/操作 | 预期结果 |
|---|---|---|---|
| TC-01 | 健康检查 | 请求健康检查接口 | 返回成功响应，服务可用。 |
| TC-02 | 用户注册唯一性 | 使用重复学号、邮箱或手机号注册 | 返回冲突或校验错误。 |
| TC-03 | 学生画像保存 | 提交专业、年级、兴趣标签 | 再次查询可看到更新后的画像。 |
| TC-04 | 赛事草稿不可见 | 管理员保存草稿，学生搜索 | 学生端不返回草稿赛事。 |
| TC-05 | 审核发布 | 管理员提交审核并通过 | 赛事状态变为已发布，学生端可见。 |
| TC-06 | 多维筛选 | 按专业、年级、类别和截止时间筛选 | 返回符合所有条件的赛事。 |
| TC-07 | 订阅提醒 | 学生订阅含未来节点的赛事 | 生成对应待发送提醒。 |
| TC-08 | 取消订阅 | 学生取消订阅 | 未发送提醒被取消，已发送消息保留。 |
| TC-09 | 推荐理由 | 学生画像与赛事标签匹配 | 推荐结果包含可解释理由。 |
| TC-10 | 权限校验 | 学生访问管理员接口 | 返回 `403 forbidden`。 |
| TC-11 | 外部链接 | 点击官方报名链接 | 明确跳转外部来源。 |
| TC-12 | 文档构建 | 运行 `just docs-build` | 文档构建成功。 |

# 4．评价

## 4.1 范围

测试范围覆盖学生端主流程、管理端发布流程、后端业务规则、权限、错误处理、提醒任务和文档构建。v0.0 不覆盖自动采集、模型推荐和多渠道通知。

## 4.2 准则

通过准则：

- 核心命令在本地和 CI 中通过。
- API 响应符合统一 envelope 和错误码约定。
- 学生端只展示可见赛事。
- 管理端状态流转和审核记录准确。
- 订阅、提醒和消息逻辑满足幂等性要求。
- 文档与功能行为一致。

## 4.3 软件能力

按设计完成后，系统应具备：

- 支持学生快速筛选和查看赛事。
- 支持管理员维护可信赛事数据。
- 支持订阅提醒和个人日历。
- 支持基于规则的可解释推荐。
- 支持持续集成和回归验证。

## 4.4 缺陷和限制

v0.0 当前限制：

- 核心业务 API 和页面交互尚待实现。
- 自动化测试仍需随业务代码补充。
- 测试数据和演示数据尚未固化。
- 联调测试需要等待接口和页面完成。

## 4.5 建议

- 每实现一个业务模块即补充对应测试，不等到项目末期集中补测。
- 对赛事状态流转、权限和提醒幂等性优先写测试。
- 前后端联调前先固定 API 示例和错误响应。
- 将常用测试数据写入 fixture 或 seed 脚本，保证演示和回归稳定。
