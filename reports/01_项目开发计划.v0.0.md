# 1．引言

## 1.1 编写目的

本文档用于说明“大学生竞赛信息智能筛选与推荐系统”（CompeteHub）的项目背景、建设目标、交付内容、实施计划、运行环境和验收标准，为课程项目开发、任务分工、进度跟踪和阶段验收提供依据。

本文档面向课程指导教师、项目团队成员、测试人员和后续维护人员。

## 1.2 项目背景

高校学生获取竞赛信息时通常需要在学校官网、学院通知、竞赛官网、社群消息等渠道之间反复检索，信息分散且格式不统一。学生在判断赛事是否适合自身专业、年级、兴趣和参赛目标时成本较高，也容易遗漏报名截止、作品提交、答辩评审等关键节点。

本项目拟建设一个以本科学生为核心用户的竞赛信息筛选与推荐系统，帮助学生完成“发现可信赛事、判断适配程度、收藏订阅赛事、跟进关键提醒”的主要任务链路。管理员负责赛事录入、审核发布、状态维护和基础运营管理。

## 1.3 定义

| 术语 | 定义 |
|---|---|
| CompeteHub | 本项目英文名称，指大学生竞赛信息智能筛选与推荐系统。 |

| 审核发布 | 管理员对赛事来源、字段完整性和展示内容进行确认后发布到学生端。 |

# 2．项目概述

## 2.1 工作内容

项目主要工作包括：

1. 产品需求梳理：明确用户角色、核心场景、功能范围和验收标准。
2. 架构设计：确定 Vue 3 前端、Flask 后端、PostgreSQL、Redis 和 Celery 的前后端分离架构。
3. 数据建模：设计用户、学生画像、赛事、时间节点、标签、收藏订阅、提醒、审核记录和审计日志等核心模型。
4. 后端开发：实现认证、用户画像、赛事查询、赛事管理、订阅提醒、推荐和审计等 API。
5. 前端开发：实现学生端赛事搜索、筛选、详情、推荐、收藏订阅、个人日历、消息中心和管理端基础页面。
6. 测试验证：建立后端单元测试、接口测试、前端 lint/build、文档构建和 CI 校验。
7. 文档交付：维护产品文档、技术文档、API 文档、测试计划、用户手册和开发记录。

## 2.2 条件与限制

已具备条件：

- 仓库已建立 monorepo 结构，包含 `apps/api`、`apps/web`、`docs`、`infra`、`reports` 和 `scripts`。
- 后端已初始化 Flask 应用工厂、配置、扩展、错误处理、健康检查和核心模型目录。
- 前端已初始化 Vue 3/Vite 应用、路由、页面占位、API client 和基础 store。
- 本地基础设施已提供 PostgreSQL 和 Redis 的 Docker Compose 配置。
- 仓库已有 `just` 命令、GitHub Actions 和 MkDocs 文档构建配置。

主要限制：

- 当前课程项目资源有限，优先实现核心链路，增强能力和后续能力按迭代推进。
- 赛事信息采集以人工录入和审核为主，自动采集不作为 v0.0 核心交付。
- 推荐能力以规则和可解释理由为主，不在 v0.0 引入机器学习模型。
- 多渠道通知先保留扩展点，v0.0 以站内提醒为主。

## 2.4 运行环境

开发环境：

- Node.js 与 npm。
- Python 3.11 与 `uv`。
- `just` 命令运行器。
- Docker Desktop 或兼容 Docker Compose 的本地环境。

运行组件：

- Web：浏览器访问 Vue SPA。
- API：Flask 服务，默认本地端口 `5000`。
- Database：PostgreSQL。
- Redis：缓存、限流、Celery broker 和短期运行状态。

## 2.5 服务

项目团队提供以下支持：

- 本地开发环境启动说明。
- 系统核心功能使用说明。
- 测试计划、测试结果记录和问题修复说明。
- 课程答辩和演示所需的数据、账号和操作流程准备。

## 2.6 验收标准

v0.0 阶段验收重点：

- 仓库结构、启动命令、文档和 CI 基础设施完整。
- 后端健康检查、配置、模型和测试基础可运行。
- 前端路由、主要页面入口和基础构建可运行。
- PRD、技术规格、API 规格和数据模型与实现方向一致。

后续核心功能验收重点：

- 学生可注册登录并维护画像。
- 学生可搜索、筛选、排序并查看赛事详情。
- 管理员可录入、审核、发布和维护赛事状态。
- 学生可收藏或订阅赛事，并看到个人赛事日历和站内提醒。
- 系统可基于学生画像和赛事标签给出规则推荐及推荐理由。

## 3.3 预算

本项目为课程实践项目，预算主要体现为人力投入和本地开发资源：

- 开发资源：团队成员本地计算机、浏览器、Docker、Node.js、Python 和开源依赖。
- 服务资源：本地 PostgreSQL 与 Redis；如需部署演示环境，可使用学校或个人云主机。
- 人力成本：按课程小组成员投入时间核算，不涉及商业采购预算。

## 3.4 关键问题

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 需求范围过大 | 核心链路无法按期完成 | 坚持核心能力优先，增强功能延后 |
| 赛事数据质量不足 | 推荐和筛选效果下降 | 强制来源字段、审核发布和状态日志 |
| 推荐解释不清晰 | 用户难以信任推荐结果 | 使用规则推荐，并展示推荐理由 |
| 前后端接口漂移 | 联调成本上升 | 以 `docs/api_spec.md` 作为接口合同 |
| 提醒任务重复或遗漏 | 用户体验受损 | 提醒以 PostgreSQL 为事实来源，异步任务保持幂等 |
