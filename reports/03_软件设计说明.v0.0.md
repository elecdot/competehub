# 1．引言

## 1.1 编写目的

本文档说明 CompeteHub 的软件设计方案，包括系统模块划分、结构设计、主要处理流程、数据库设计、接口设计、界面设计和出错处理策略，为后续编码、测试和维护提供依据。

## 1.2 项目背景

CompeteHub 面向本科学生提供可信竞赛信息检索、筛选、推荐、订阅和提醒服务，并为管理员提供赛事录入、审核发布和状态维护能力。系统采用前后端分离架构，核心业务数据以 PostgreSQL 为事实来源。

## 1.3 定义

| 术语 | 定义 |
|---|---|
| SPA | 单页应用，指基于 Vue 3 构建的 Web 前端。 |
| REST API | 前后端通过 HTTP/JSON 交互的接口形式。 |
| Blueprint | Flask 中用于组织 API 路由的模块化机制。 |
| Service | 后端业务服务层，负责状态流转和跨表规则。 |
| Repository | 后端数据访问层，封装数据库查询。 |

# 2．概要设计

## 2.1 系统模块划分

系统按用户工作流和后端领域划分模块。

学生端模块：

- 首页与导航模块：提供主要入口。
- 赛事列表模块：关键词搜索、多维筛选、排序和分页。
- 赛事详情模块：展示完整赛事信息、时间节点、适配标签、来源和官方通道。
- 推荐模块：展示规则推荐列表和推荐理由。
- 收藏订阅模块：管理收藏、订阅和提醒配置。
- 个人日历模块：展示已订阅赛事节点。
- 消息中心模块：展示站内提醒和已读状态。
- 个人画像模块：维护专业、年级、兴趣和偏好。

管理端模块：

- 赛事管理模块：赛事录入、编辑、草稿、提交审核和状态维护。
- 审核模块：待审核赛事处理、审核意见和记录。
- 基础配置模块：赛事类别、标签、专业、年级和提醒节点配置。
- 用户权限模块：用户状态、角色和权限管理。
- 日志统计模块：审核记录、操作日志和简单运营统计。

后端模块：

- `auth`：注册、登录、登出、当前用户。
- `users`：学生画像和偏好。
- `competitions`：学生端赛事列表、筛选、详情和外链记录。
- `admin`：赛事录入、审核、状态管理和配置。
- `subscriptions`：收藏、订阅和个人日历。
- `notifications`：站内消息、提醒设置和已读状态。
- `recommendations`：规则推荐和推荐理由。
- `audit`：操作日志查询。

## 2.2 系统结构设计

系统采用前后端分离的 monorepo 结构：

```text
Browser
  -> Vue 3 SPA (apps/web)
  -> Flask API /api/v1 (apps/api)
  -> PostgreSQL
  -> Redis
  -> Celery worker
```

后端请求处理分层：

```text
routes -> schemas -> services -> repositories -> models
```

各层职责：

- routes：处理 HTTP 参数、认证上下文和响应包装。
- schemas：请求校验和响应序列化。
- services：业务规则、状态流转和跨表一致性。
- repositories：数据库查询封装。
- models：表结构、枚举和关系。
- tasks：异步任务入口，调用 services。

## 2.3 处理流程设计

### 2.3.1 赛事发布流程

```text
管理员创建草稿
  -> 填写来源、时间节点、条件、标签和正文
  -> 提交审核
  -> 审核人核对信息
  -> 通过后状态变为已发布
  -> 学生端搜索、列表、详情和推荐可见
```

驳回、下架、归档、取消和过期均需记录原因和操作日志。

### 2.3.2 学生订阅提醒流程

```text
学生查看赛事详情
  -> 收藏或订阅赛事
  -> API 创建订阅记录
  -> 根据赛事时间节点生成待发送提醒
  -> Celery 扫描到期提醒
  -> 生成站内消息
  -> 学生在消息中心或个人日历查看
```

取消订阅时，未来未发送提醒应被取消，已发送消息保留。

### 2.3.3 规则推荐流程

```text
读取学生画像
  -> 读取已发布赛事及标签
  -> 按专业、年级、兴趣、截止时间和热度评分
  -> 生成排序结果
  -> 为每条赛事生成推荐理由
```

画像不足时降级为近期截止、热门或最新发布赛事。

# 3．数据库设计

核心表设计如下：

| 表 | 说明 |
|---|---|
| `users` | 账号、角色、状态和基础身份信息。 |
| `student_profiles` | 学生画像、兴趣、竞赛经历和提醒偏好。 |
| `competitions` | 赛事主表，包含来源、标题、类别、条件、详情和状态。 |
| `competition_time_nodes` | 赛事关键时间节点。 |
| `competition_tags` | 标签字典。 |
| `competition_tag_links` | 赛事与标签关系。 |
| `favorites` | 收藏关系。 |
| `subscriptions` | 订阅关系和提醒覆盖配置。 |
| `reminder_settings` | 用户默认提醒设置。 |
| `reminders` | 提醒计划、触发时间和发送状态。 |
| `messages` | 站内消息和已读状态。 |
| `review_records` | 赛事审核记录。 |
| `audit_logs` | 操作日志。 |
| `recommendation_rules` | 推荐规则和理由模板。 |
| `system_configs` | 系统基础配置。 |

主要约束：

- 用户邮箱、手机号和学号在存在时分别唯一。
- 赛事来源名称和来源链接必填。
- 草稿、待审核、驳回和下架赛事默认不对学生端可见。
- 收藏和订阅独立，取消收藏不应取消订阅。
- 提醒状态变更必须幂等，避免重复生成消息。

# 4．接口设计

## 4.1 外部接口

系统外部接口主要是官方赛事链接和学校、学院通知来源链接。系统只保存来源和跳转入口，不伪造官方报名结果。

## 4.2 内部接口

所有应用接口使用 `/api/v1` 前缀，响应采用统一 envelope：

```json
{
  "data": {},
  "error": null
}
```

错误响应：

```json
{
  "data": null,
  "error": {
    "code": "validation_error",
    "message": "请求参数不合法",
    "details": {}
  }
}
```

主要接口组：

- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/logout`
- `GET /me`
- `GET /me/profile`
- `PATCH /me/profile`
- `GET /competitions`
- `GET /competitions/{id}`
- `POST /competitions/{id}/favorite`
- `POST /competitions/{id}/subscribe`
- `GET /me/calendar`
- `GET /me/messages`
- `GET /recommendations`
- `POST /admin/competitions`
- `POST /admin/competitions/{id}/submit-review`
- `POST /admin/competitions/{id}/approve`
- `POST /admin/competitions/{id}/reject`

# 5．模块详细设计

## 5.1 后端模块

后端使用 Flask application factory 初始化应用。配置通过环境变量加载，测试环境使用内存 SQLite，开发环境默认可连接本地数据库。扩展集中在 `extensions.py` 初始化。

业务代码按以下结构演进：

```text
blueprints/
models/
repositories/
schemas/
services/
tasks/
```

服务层负责状态流转，例如赛事审核、订阅生成提醒、提醒发送和推荐评分。异步任务通过 Celery 调用服务层，避免复制业务规则。

## 5.2 前端模块

前端使用 Vue Router 管理页面路由，Pinia 管理跨页面状态。页面按工作流组织：

- `CompetitionListPage`：赛事列表。
- `CompetitionDetailPage`：赛事详情。
- `RecommendationPage`：推荐赛事。
- `CalendarPage`：个人赛事日历。
- `AdminHomePage`：管理端入口。

状态 store 包括登录状态、学生画像、赛事筛选条件、基础字典和通知状态。

## 5.3 异步任务模块

Celery worker 负责：

- 扫描到期提醒。
- 生成站内消息。
- 标记过期赛事。
- 后续扩展半自动采集候选赛事。

Redis 用于任务 broker、结果后端、缓存和短期锁，不保存唯一业务事实。

# 6. 界面设计

## 6.1 界面样式设计

界面以清晰、克制、便于扫描为原则。学生端强调快速查找和判断，管理端强调信息录入、审核和状态确认。

学生端主要页面：

- 首页：系统入口、搜索入口和推荐入口。
- 赛事列表：搜索框、筛选区、排序、结果列表和分页。
- 赛事详情：来源、时间节点、条件、标签、官方通道、收藏订阅按钮。
- 推荐赛事：推荐卡片和理由说明。
- 我的日历：按日期展示已订阅节点。
- 消息中心：提醒消息和已读状态。

管理端主要页面：

- 赛事管理：列表、状态筛选和编辑入口。
- 赛事表单：结构化字段和校验提示。
- 审核页面：详情核对、通过、驳回和意见填写。

## 6.2 界面交互设计

- 筛选条件应可叠加、可清除，并实时反馈结果数量。
- 订阅操作应在 2 步以内完成。
- 推荐理由应随赛事一起展示，避免用户无法理解排序依据。
- 管理端表单对必填字段、链接格式和时间冲突进行即时提示。
- 外部官方链接跳转应明确标识来源。

# 7．出错处理设计

## 7.1 错误类型及出错处理对策

| 错误类型 | 示例 | 处理方式 |
|---|---|---|
| 参数校验错误 | 缺少标题、链接格式错误 | 返回 `400 validation_error`，提示具体字段。 |
| 未登录 | 访问个人中心 | 返回 `401 unauthorized`，前端引导登录。 |
| 权限不足 | 学生访问管理端接口 | 返回 `403 forbidden`。 |
| 资源不可见 | 访问未发布赛事 | 返回 `404 not_found` 或按权限隐藏。 |
| 状态冲突 | 已审核赛事重复提交 | 返回 `409 conflict`。 |
| 外部服务异常 | Redis 或任务队列暂不可用 | 保留数据库事实，提示稍后重试并记录日志。 |
| 未知错误 | 未处理异常 | 返回 `500 internal_server_error`，记录服务端日志。 |

前端应统一处理错误响应，避免直接暴露后端堆栈信息。
