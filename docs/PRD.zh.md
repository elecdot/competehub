# 大学生竞赛信息智能筛选与推荐系统 PRD

## 一、文档目的

本文档用于将“大学生竞赛信息智能筛选与推荐系统”从概念性功能清单细化为面向开发、测试和后续排期讨论的产品需求。为避免 PRD 在后续实现中漂移或过期，本文档优先承担稳定功能边界、业务对象和设计约束的作用，不作为逐项严格验收合同。后续实现可按核心能力、增强能力和后续能力分层交付，但不应改变本文档中已经明确的主要业务语义。

## 二、产品背景与目标

### 2.1 背景问题

在校本科生获取竞赛信息时通常面临以下问题：

- 赛事信息分散在学校官网、学院通知、竞赛官网、社团群聊等多个渠道，学生需要反复查找。
- 不同竞赛的报名条件、截止时间、参赛形式和材料要求表达不统一，筛选成本高。
- 学生难以判断某项竞赛是否适合自己的专业、年级、兴趣和参赛目标。
- 报名、作品提交、答辩等关键节点容易被遗漏。
- 往届经验、资料和复盘缺少统一沉淀，后续参赛者难以复用。

### 2.2 产品目标

系统优先服务本科学生，帮助其更高效地完成“找到可信竞赛、判断是否适合、订阅关键节点、持续跟进提醒”的完整任务链路。当前正式角色为学生和管理员：学生负责发现、判断和跟进赛事，管理员负责赛事信息维护、审核发布、规则配置和运营统计。教师、教学秘书、竞赛组织者和学生社团作为辅助干系人，其诉求用于指导后续扩展，但当前不单独建设独立工作台。

### 2.3 成功标准

产品成功标准以学生任务效率和信息可信度为主：

- 学生能够在 3 次以内筛选操作中定位一组符合专业、年级、兴趣或时间条件的赛事。
- 学生能够在赛事详情页明确看到来源、关键时间节点、报名条件、参赛形式、适配标签和官方通道。
- 学生能够在 2 步以内完成收藏或订阅，并在个人中心或日历中看到已订阅赛事。
- 订阅赛事的报名截止、作品提交、比赛开始等关键节点能够生成站内提醒。
- 管理员发布的赛事必须经过审核状态流转，并保留来源和操作日志。

## 三、用户与场景

### 3.1 核心用户

本科学生是系统的核心用户。其主要目标是用较低成本发现适合自己的竞赛，并减少错过报名和提交节点的风险。

当前 P1 采用单一高校部署边界：学生、学号、学院、管理员、基础字典和校级治理均属于部署高校。系统可以收录全国组织或其他高校可信来源中的赛事，但不因此成为多高校共享租户平台。

### 3.2 管理角色与辅助干系人

- 系统管理员：负责账号权限、赛事录入审核、内容审核、规则配置和运营统计。
- 教师或教学秘书：关注赛事信息是否准确、是否便于推荐给学生，以及学生参与情况。
- 竞赛组织者或学生社团：关注赛事宣传、组队招募、答疑和资料沉淀。

### 3.3 典型场景

- 学生按专业、年级、兴趣标签和截止时间查找近期可报名竞赛。
- 学生查看某项竞赛详情，判断其报名条件、参赛形式、时间安排和适配程度。
- 学生收藏或订阅赛事，并在个人赛事日历中跟进报名截止和作品提交节点。
- 系统根据学生画像和赛事标签，提供规则推荐及推荐理由。
- 管理员从可信来源人工录入赛事信息，审核通过后发布到前台。

## 四、范围分层与交付原则

本文档尽量完整写明系统可预见的功能需求，并通过分层方式表达交付顺序建议。核心能力优先支撑主任务链路，增强能力和后续能力保留扩展空间；具体排期可随项目资源调整，但应保持数据模型、权限边界和用户体验的一致性。

### 4.1 核心能力

核心能力是当前 PRD 重点细化的范围，用于支撑学生完成主要任务链路：

- 用户注册登录与学生画像维护。
- 管理员人工录入赛事、审核发布和状态管理。
- 赛事搜索、筛选、排序和列表展示。
- 赛事详情、适配标签、价值依据说明和官方通道跳转。
- 收藏订阅、站内提醒和个人赛事日历。
- 基于规则的个性化推荐和推荐理由展示。

### 4.2 增强能力

增强能力用于提升体验和运营效率，后续可根据排期逐步细化：

- 推荐偏好调整和不感兴趣反馈。
- 赛事资料归档、常见问题、往届经验和复盘内容。
- 教师或教秘辅助查看学生关注热度、赛事覆盖情况和推荐效果。
- 半自动采集候选赛事，但仍需人工审核后发布。

### 4.3 后续能力

后续能力保留为产品演进方向，不作为当前核心链路的必要前提：

- 论坛、评论、点赞、收藏、组队交流和认证答疑。
- 多渠道提醒，例如邮件、短信、微信或企业微信。
- 基于机器学习模型的推荐。
- 更完整的数据看板、运营分析和跨学院管理。

### 4.4 文档维护原则

- PRD 作为产品边界和设计约束文档，需求细节发生实质变化时应同步更新。
- 迭代验收应以对应开发任务、测试用例或阶段计划为准，PRD 提供需求来源和一致性依据。
- 新增能力应优先复用既有业务对象、状态流转和权限模型，避免为短期实现引入割裂设计。

## 五、业务对象与状态

### 5.1 用户画像

用户画像用于搜索筛选、规则推荐和统计分析。主要字段包括：

- 基础信息：用户 ID、姓名或昵称、角色、账号状态，以及独立绑定的类型化学号、邮箱或手机号账号标识。
- 学生画像：专业、年级、学院、兴趣方向、竞赛经历、目标偏好。
- 偏好信息：关注类别、屏蔽类别，以及由独立提醒设置维护的默认提前天数、节点类型和全局站内提醒开关。
- 画像状态：根据当前字段动态计算为 `incomplete` 或 `recommendation_ready`。后者要求学院、专业、年级和至少一个兴趣标签有效完整；不得持久化独立完成布尔值。

### 5.2 赛事信息

赛事信息应来自可信来源，并由管理员录入或维护。领域上区分长期稳定的赛事系列与一次具体举办周期的赛事届次。每个年度或独立参与周期创建新的赛事届次并关联原赛事系列；同一届内的延期或截止时间修改更新原届次，不得覆盖往届记录。当前搜索、详情、收藏、订阅、审核和提醒均以赛事届次为对象。

主要字段包括：

- 身份字段：赛事系列 ID、赛事届次 ID、来源支持的年度或届次标识。
- 基础字段：标题、简称、类别、主办方、承办方、赛事级别或参考标签。
- 来源字段：来源名称、来源链接、官方报名链接、附件链接、录入人。
- 适配字段：专业适配范围、年级适配范围、适合专业、适合年级、可同时包含个人与团队的参赛形式、团队人数、报名条件、材料要求。专业和年级范围必须明确为全部适用、指定范围或资料未知，不使用空列表表达多种含义。
- 时间节点：按赛事阶段组织报名开始、报名截止、作品提交截止、比赛开始、比赛结束、答辩或评审时间、结果公布时间；其他节点必须提供面向用户的说明，且默认不参与报名筛选和提醒。相关的开放/截止或开始/结束节点成对展示但分别存储，节点使用 `primary` 或 `secondary` 重点级别控制展示层级。同届次改期保留稳定逻辑节点标识；每个赛事内容修订保存独立不可变节点快照，行为事实变化时才递增节点修订号。
- 展示字段：简介、详情正文、适配说明、价值依据说明、注意事项。
- 届次生命周期状态：未发布、已发布、已下架、已归档、已取消、已过期。
- 修订状态：草稿、待审核、已通过、已驳回、已退回。草稿和待审属于修订，不是赛事届次生命周期。

### 5.3 收藏与订阅

收藏与订阅是两个独立的学生行为。收藏用于保存赛事以便后续查看；订阅用于持续跟进赛事时间节点，并可触发站内提醒和个人赛事日历节点。主要字段包括：

收藏与订阅均绑定一个具体赛事届次，不随同系列的新届次自动继承。未来若增加“关注赛事系列”，它只用于通知新届次发布，不得在缺少学生再次确认时自动创建届次订阅或截止提醒。

- 用户 ID、赛事 ID、收藏状态、订阅状态。
- 提醒配置：是否开启提醒、提前提醒天数、提醒节点范围。
- 记录状态：有效、已取消、赛事下架后失效。

### 5.4 站内提醒

站内提醒用于系统内消息中心和个人赛事日历。主要字段包括：

- 提醒 ID、用户 ID、赛事 ID、节点类型、触发时间、提醒标题、提醒内容。
- 提醒计划状态：待发送、已发送、已取消、发送失败。已读/未读属于投递后的站内消息，不属于提醒计划状态。

### 5.5 审核记录

审核记录用于保存赛事修订和推荐规则集版本的不可变审核决定；关键后台操作另写不可变审计事件。主要字段包括：

- 审核对象及版本、提交人、审核人、审核状态、审核意见、差异、影响摘要、决定时间。
- 审计事件记录操作者、受控动作、目标、结果、动作级允许详情和发生时间；创建、修改、提交审核、通过、驳回、退回、下架、归档和取消均产生对应事件。

### 5.6 推荐规则

推荐规则用于生成可解释的赛事推荐。主要字段包括：

- 匹配条件：专业、年级、兴趣标签、赛事类别、受控参考标签、报名状态和截止时间。
- 内部排序权重：专业匹配、年级匹配、兴趣匹配、报名可行动性和时间紧迫度；权重属于独立版本化规则集，不作为公开赛事分数。
- 推荐理由：与专业匹配、与兴趣匹配、适合当前年级、报名中或即将开始、近期截止，以及由受控规则和公开赛事事实支持的通用理由。P2 不使用收藏热度生成排序或理由。

## 六、功能需求

### FR1 用户注册登录与画像维护

#### FR1.1 注册与登录

- 用户故事：作为本科学生，我希望使用系统当前开放且已经验证的账号标识登录，以便保存个人画像、收藏订阅和提醒信息。
- 前置条件：用户未登录；系统可校验账号唯一性；密码规则已配置；公开注册入口仅在对应的真实验证发送渠道可用时开放。P1 单因素密码至少 15 个、最多 128 个 Unicode 字符，支持空格和密码管理器，不设置字符种类组合规则。
- 主流程：P1 只在配置可用邮件发送方时开放邮箱自助注册。学生提交邮箱和密码后，系统创建 `pending_activation` 账号并发送单次、限时验证码，但不创建登录态；验证成功后账号变为 `active`，学生再通过常规登录进入首页或上一访问页面。未配置邮件发送方时不展示公开注册入口，学生使用高校名册、管理员预置或受控邀请路径获得账号。
- 异常流程：注册和重发接口使用不泄露账号是否存在的一致响应；验证码错误、过期或尝试次数超限时拒绝激活；密码过短、过长或命中本地常见、已泄露及上下文弱密码阻止列表时给出可行动的修改提示；待激活或被禁用账号不得登录。若生产环境启用了邮箱注册但未配置发送方，服务必须拒绝启动，而不是返回或记录验证码。
- 验收标准：同一类型和规范化值不能绑定多个有效账号；创建账号不会自动登录；只有 `active` 账号可以创建个人画像、收藏、订阅和提醒数据；登录成功后可访问个人中心；未登录用户访问个人功能时应被引导登录。
- 标识规则：账号主体与账号标识分离，一个用户可绑定多个类型化标识；登录不得跨类型搜索同一文本或依靠密码消歧，失败响应不得泄露标识是否存在。P1 数据模型支持学号、邮箱和手机号，但只开放已配置的邮箱自助注册；学号由部署高校名册或管理员受控配置，手机号短信验证与公开注册延后。
- 权限发现：当前用户响应返回角色和受控 capability 列表，学生列表为空；前端据此发现可进入的管理员工作台，但后端仍对每个接口独立鉴权。

#### FR1.2 学生画像维护

- 用户故事：作为本科学生，我希望维护专业、年级、兴趣方向和目标偏好，以便系统提供更准确的筛选默认项和推荐结果。
- 前置条件：用户已登录、账号为 `active` 且角色为学生。账号激活后存在一条可编辑的 `incomplete` 画像，但首次进入不强制填写。
- 主流程：学生进入个人中心；可以跳过或编辑学院、专业、年级、兴趣方向、竞赛经历和目标偏好；系统根据当前值动态返回画像状态和缺失字段。学院、专业、年级及至少一个兴趣标签有效时进入 `recommendation_ready`，后续推荐才使用个性化匹配。
- 异常流程：学院、专业、年级或标签不在部署高校受控字典中，专业不属于所选学院，或兴趣标签超过 10 个时提示修正并保留已填写内容；保存失败时提示重试。画像不足不阻止搜索、详情、收藏、订阅或提醒。
- 验收标准：保存成功后再次进入个人中心可看到最新画像、`profile_status` 和 `missing_fields`；完整画像可产生可追溯的个性化推荐理由；不完整画像使用通用可行动推荐并明确降级依据；竞赛经历、目标偏好、屏蔽标签和显示名可清空且不影响画像完成状态。

### FR2 赛事人工录入、审核与状态管理

#### FR2.1 管理员人工录入赛事

- 用户故事：作为管理员，我希望从可信来源人工录入赛事信息，以便学生看到结构统一、来源明确的竞赛内容。
- 前置条件：管理员已登录后台；系统已配置赛事类别、级别或参考标签、适合专业、适合年级等基础字典。
- 主流程：管理员进入赛事管理；新建赛事；填写标题、来源、主办方、类别、时间节点、报名条件、官方链接、附件和适配信息；保存为草稿或提交审核。
- 异常流程：必填字段缺失时禁止提交审核；当前 P1 若没有至少一个类型受控且带唯一有效发生时刻的赛事时间节点，也不得提交审核；官方链接格式错误时提示修正；报名截止早于报名开始时提示时间冲突；相似标题和主办方已存在时提示可能属于同一赛事系列或重复届次，由管理员根据来源事实确认，不自动合并。
- 验收标准：草稿不在前台展示；提交审核后赛事状态为待审核；赛事必须保留来源名称和来源链接；重复提示不阻止管理员保存草稿，但需在审核页展示。
- 发布完整度：提交审核至少需要赛事系列、来源支持的届次标识、标题、来源名称、合法 HTTP(S) 来源链接、类别、主办方、简介、报名条件、至少一种参赛形式、至少一个赛事阶段和一个带发生时刻的 `primary` 核心节点。允许团队参赛时必须填写团队人数；专业和年级适配范围必须显式填写。简称、承办方、详情长文、标签、价值依据、官方报名链接、附件和非核心节点可以缺失，提供的链接仍必须通过 HTTP(S) 校验。
- 管理界面：P1 必须提供真实赛事治理工作台，支持赛事系列/届次选择、来源与发布字段、阶段和成对节点编辑、重点级别、完整度反馈、保存草稿和提交审核。CLI、seed 和 API-only 验收只能作为实现中间里程碑。
- 查询支撑：工作台 API 必须支持系列搜索、届次/修订筛选、单届次 workspace、全局待审队列、修订完整内容、发布完整度、基于不可变基线的字段/阶段/节点差异和订阅提醒影响预览；终态审核记录不能代替待审查询。

#### FR2.2 赛事审核发布

- 用户故事：作为管理员，我希望审核待发布赛事，以便保证前台展示的信息可信、完整且不误导学生。
- 前置条件：存在待审核赛事；审核人具备审核权限。
- 主流程：审核人查看待审核列表；进入详情核对来源、时间节点、链接、适配标签和正文；选择通过或驳回；填写审核意见；系统记录审核结果；通过后赛事进入已发布状态并前台可见。
- 异常流程：审核人发现来源不可信时驳回并说明原因；发现字段缺失时驳回或退回修改；赛事已被其他管理员处理时提示状态已变化并刷新；当前修订由本人提交时禁止自审，提交者可在尚未审核前撤回到草稿。
- 验收标准：只有已发布赛事可在学生端搜索、列表和推荐中出现；每次审核操作均记录审核人、时间、状态和意见；已驳回赛事不在前台展示。
- 职责分离：赛事编辑与赛事审核是管理员权限而非新角色；同一账号可拥有两项权限，但当前修订的审核者必须不同于提交者。
- 修订发布：赛事届次内容通过编号修订保存。已发布后修改时创建新草稿修订，学生继续读取上一公开修订；独立审核通过后原子切换公开修订。已提交和已审核修订不可改写，审核界面必须展示差异与影响预览。
- 并发边界：P1 每个赛事届次最多存在一个 `draft` 或 `pending_review` 进行中修订。替换修订记录其复制的 `base_revision_id`；审批锁定届次并校验公开修订仍等于该基线，否则返回 `409 stale_revision`，不写终态审核决定。节点修订号由服务端在提交时相对基线冻结。
- 审核交互：待审工作台必须展示提交者、来源事实、字段/阶段/节点差异、公开与提醒影响，并提供填写意见的通过、驳回和退回操作；自审在界面和 API 两层阻止。

#### FR2.3 赛事状态管理

- 用户故事：作为管理员，我希望维护赛事状态，以便已截止、取消、下架或归档的赛事不会误导学生。
- 前置条件：赛事已存在；管理员具备赛事管理权限。
- 主流程：管理员在赛事列表筛选目标赛事；修改状态为已下架、已归档、已取消或已过期；填写操作原因；系统同步更新学生端展示和订阅提醒。
- 异常流程：已取消或已下架赛事仍有订阅用户时，系统取消后续未发送提醒并保留已发送消息；归档或过期仅在当前公开修订不存在未来节点时允许，否则返回冲突并要求修正时间事实，或按真实情况使用取消/紧急下架。归档、过期保留收藏、订阅历史关系和过去日历节点，事务性取消异常残留的待发送提醒，但不生成新的站内消息。
- 验收标准：已下架赛事不在学生端列表、搜索和推荐中展示；已取消赛事详情页展示取消提示；状态变更必须写入操作日志。
- 紧急下架：当前公开内容存在安全、欺诈、链接劫持或严重误导风险时，具备赛事维护权限（`competition_maintainer`）的管理员可填写原因后立即下架；该权限不包含修订编辑或审核，恢复公开必须提交修正修订并由另一审核者批准，不允许直接改回已发布。
- 状态交互：治理工作台提供取消、归档、过期和紧急下架入口，展示对公开可见性、订阅、提醒、日历和消息的影响；归档/过期预览必须列出阻塞转换的未来节点和异常残留提醒数，紧急下架必须在确认操作中填写原因。

### FR3 赛事搜索、筛选与列表展示

#### FR3.1 关键词搜索

- 用户故事：作为本科学生，我希望用赛事名称、主办方或关键词搜索赛事，以便快速找到目标竞赛。
- 前置条件：系统存在已发布赛事；用户可以未登录访问基础搜索。
- 主流程：学生在搜索框输入关键词；系统在赛事标题、简称、主办方、类别和正文摘要中检索；返回匹配赛事列表；学生可进入详情页。
- 异常流程：关键词为空时展示默认推荐或最新赛事；无结果时提示调整关键词或清空筛选条件；搜索服务异常时提示稍后重试。
- 验收标准：搜索结果只包含已发布且未下架赛事；列表展示标题、类别、主办方、关键时间节点、参考标签和收藏入口；无结果状态必须有明确提示。

#### FR3.2 多维筛选

- 用户故事：作为本科学生，我希望按专业、年级、赛事类别、参考标签、报名状态、报名截止日期和参赛形式筛选，以便减少无关赛事。
- 前置条件：赛事已配置对应筛选字段；筛选项字典可用。
- 主流程：学生打开赛事列表；选择一个或多个筛选条件；系统刷新结果；学生可继续叠加或移除条件。
- 异常流程：筛选组合无结果时提示清除部分条件；筛选字典加载失败时保留基础列表并提示部分筛选不可用。
- 验收标准：多条件筛选采用交集逻辑；当前筛选条件在页面上清晰展示且可一键清除；筛选结果数量随条件变化更新。
- 报名状态：报名状态从当前公开修订的报名阶段和节点动态计算为报名中、即将开始、已截止、未知或不适用。多轮赛事优先返回报名中的状态，其次即将开始；“不适用”必须由管理员显式填写，不从缺少节点推断。公开筛选不得把赛事发布状态伪装成报名状态。

#### FR3.3 排序与分页

- 用户故事：作为本科学生，我希望按截止时间、推荐度、发布时间或热度排序，以便根据当前目标浏览赛事。
- 前置条件：搜索或筛选结果已生成；排序字段存在。
- 主流程：学生选择排序方式；系统按指定规则返回分页结果；学生翻页或加载更多。
- 异常流程：公开列表不提供推荐度或无可靠数据支撑的热度排序；排序字段为空时使用可行动排序，排序参数不支持时返回明确校验错误。
- 验收标准：截止时间排序应优先展示未截止赛事；分页或加载更多不能重复展示同一赛事；切换排序后保留当前筛选条件。
- 默认排序：依次展示报名中、即将报名、报名状态未知、报名不适用和已截止届次；组内使用相关报名或下一个重点节点时间，并以公开时间倒序、赛事届次 ID 倒序稳定打破并列。显式支持报名截止和最新公开排序，切换后回到第一页。

### FR4 赛事详情、适配标签与官方通道

#### FR4.1 赛事详情展示

- 用户故事：作为本科学生，我希望在详情页看到完整赛事信息，以便判断是否报名或订阅。
- 前置条件：赛事处于已发布、已取消、已过期或已归档等可查看状态。
- 主流程：学生从列表或推荐进入详情页；系统展示赛事简介、来源、主办方、报名条件、时间节点、参赛形式、材料要求、附件、适配标签、注意事项和官方链接。
- 异常流程：赛事已下架时提示内容不可访问；官方链接缺失时隐藏跳转入口并展示说明；附件失效时提示以官方页面为准。
- 验收标准：详情页必须展示来源标识和更新时间；关键时间节点按时间顺序展示；已取消、已过期或已归档状态必须有明显提示。
- 时间交互：详情按赛事阶段分组展示节点，成对节点同组呈现，突出当前阶段和下一个重点节点；列表优先展示最近的未来 `primary` 节点，没有时回退到普通节点。

#### FR4.2 适配标签与价值依据说明

- 用户故事：作为本科学生，我希望看到赛事为什么适合我或值得关注，以便降低判断成本。
- 前置条件：赛事已配置类别、参考标签、适合专业、适合年级和价值依据；用户画像可选。
- 主流程：系统根据赛事配置展示参考标签，例如“校级推荐”“国家级赛事”“适合低年级”“团队赛”；根据用户画像展示适配说明；展示主办单位、认可信息、往届关注度等依据。
- 异常流程：用户未登录或画像不足时只展示赛事通用标签；学院认可信息缺失时不展示推断性结论；信息不确定时标注“以官方通知为准”。
- 验收标准：系统不得使用百分制或绝对分数表达赛事价值；参考标签必须能追溯到配置字段或官方来源；详情页必须说明该信息仅作选赛参考，不替代学校或官方认定。

#### FR4.3 官方通道跳转

- 用户故事：作为本科学生，我希望从详情页跳转到官方报名、通知或附件页面，以便完成报名和材料下载。
- 前置条件：赛事已配置一个或多个官方通道链接。
- 主流程：学生点击官方报名、通知原文或附件链接；浏览器直接打开当前可查看公开修订中的 HTTP(S) 目标，同时尽力发送非阻断点击事件。新窗口使用 `noopener/noreferrer`，统计接口不代理或重定向目标地址。
- 异常流程：链接为空或当前不可访问时不展示入口；统计请求失败、超时或被浏览器拦截时仍完成跳转；外部站点异常时提示以官方渠道恢复为准；统计接口对非法目标类型、来源页面和过量请求进行拒绝或限速。
- 验收标准：跳转入口必须标明目标类型和外部站点；系统只记录赛事届次、公开修订、`source_url`/`official_url`/`attachment_url` 目标类型、受控来源页面、服务端发生时间和登录状态类别，不保存用户 ID、账号标识、IP、User-Agent 或跨日访客标识；数据只能表述为站内记录的外链点击次数，不能表述为人数或报名转化。
- 数据保留：原始点击事件保留 90 天，随后只保留按上海产品日历日期、赛事届次、目标类型、来源页面和登录状态类别聚合的计数。浏览器或网络限制可能造成漏记，管理端必须展示该口径说明。

### FR5 收藏订阅、站内提醒与个人赛事日历

#### FR5.1 收藏与订阅

- 用户故事：作为本科学生，我希望收藏或订阅感兴趣的赛事，以便后续持续跟进。
- 前置条件：用户已登录；赛事可见且未下架。
- 主流程：学生在列表或详情页点击收藏时只创建收藏。点击订阅时，首次展示确认面板并明确列出是否开启站内提醒、提前天数和该届次现有的 `primary` 核心节点类型；界面可预填全局默认的开启、提前 3 天和推荐节点类型，但学生确认后才创建订阅及提醒计划。后续订阅可使用已保存默认值，并在结果提示中展示实际提醒配置及撤销/设置入口。
- 异常流程：未登录用户点击时引导登录；赛事已过期时允许收藏但提示提醒节点可能不可用；重复订阅时提示已订阅并保持原配置；没有仍可安排的未来触发时刻时保留订阅和日历关系，明确提示当前未安排提醒。
- 验收标准：收藏和订阅状态在列表、详情页和个人中心保持一致；收藏永不创建提醒；订阅可以在提醒关闭时保持有效并继续显示于个人日历；取消订阅后不再生成新的提醒；收藏不等同于订阅，二者状态可独立管理。
- 生命周期交互：`published` 可新增收藏、订阅和修改订阅设置；`cancelled`、`archived`、`expired` 仍可新增收藏但不能新增或修改订阅；`offline` 和从未发布的届次不能新增两类关系。无论当前状态和详情是否可访问，owner 始终可幂等删除既有收藏或订阅，避免个人状态被锁死。
- 跨届规则：新赛事届次发布后，旧届次的收藏和订阅保持原记录，但不会自动复制到新届次。

#### FR5.2 站内提醒

- 用户故事：作为本科学生，我希望系统在关键时间节点前通过站内消息提醒我，以便减少错过报名和提交的风险。
- 前置条件：用户已订阅赛事并在确认时开启提醒；全局站内提醒开关开启；赛事存在仍可安排的未来节点和触发时刻。
- 主流程：每项订阅使用一个 `0–30` 天提前量和受控节点类型集合，根据对应未来节点生成待发送提醒；到达触发时间后生成站内消息。P1 同一节点只安排一次普通提醒。
- 异常流程：赛事下架、取消或节点被删除时取消未发送提醒；节点时间修改后保留逻辑节点身份和不可变快照历史，取消基于旧快照的未发送提醒并按新时间重建未来提醒；全局开关关闭时取消全部未发送计划但保留订阅，重新开启时只重建仍在未来的计划；触发时刻已经过去时不得立即补发伪准时提醒。瞬时投递失败记录受控错误与下次重试时间，由重试任务先恢复为待发送后再投递；永久失败或耗尽重试后保持失败状态。
- 验收标准：可选提醒节点至少覆盖报名截止、作品提交截止和比赛开始三类受控核心节点；提醒内容包含赛事名称、节点类型、发生时间和详情入口；订阅响应明确实际配置、安排数量、下次提醒时刻或未安排原因；每个批准修订对受影响订阅者最多生成一条幂等的赛程变更汇总消息，只有发生时刻、所选节点增删或所选节点类型变化才触发。阶段、重点级别、描述、标题等展示修正只刷新当前日历和未发送提醒内容，不伪装成改期；已发送消息不得被改写或删除。

#### FR5.3 站内消息中心

- 用户故事：作为本科学生，我希望在一个固定入口查看历史和未读站内消息，以便提醒投递、赛事改期、取消或下架不会静默消失。
- 前置条件：用户已登录；系统已为该用户投递至少一条消息，或消息中心展示空状态。
- 主流程：全局导航消息图标展示未读数；学生进入消息中心，在全部和未读视图中按类型筛选、分页浏览，打开单条消息或执行全部已读。消息按创建时间倒序稳定排列，并保留当时的赛事标题、节点时间、事件原因摘要和目标引用快照。
- 异常流程：消息关联赛事已下架或不可访问时仍展示历史快照，并将目标入口标为不可访问；重复 worker 或重复状态事件不得生成重复消息；空列表提供返回赛事发现的入口。
- 验收标准：P1 只生成 `reminder_due`、`competition_time_changed`、`competition_cancelled` 和 `competition_offline` 四类消息；单条已读和全部已读均幂等更新未读数；已读消息不删除，正文快照不可改写；消息保留 365 天，P1 不提供逐条删除，账号删除时按账号数据策略清理。
- 停止提醒规则：用户主动取消订阅或关闭提醒时只反馈操作结果，不生成历史消息；赛事取消或紧急下架导致未来提醒停止时，为每个有效订阅者和领域事件生成一条幂等消息。

#### FR5.4 个人赛事日历

- 用户故事：作为本科学生，我希望在个人赛事日历中查看已订阅赛事节点，以便安排参赛计划。
- 前置条件：用户已登录且存在至少一个订阅赛事；收藏本身不进入个人赛事日历。
- 主流程：学生进入个人赛事日历；系统按 `Asia/Shanghai` 日期展示订阅选择的全部节点，提供月、周和列表三种视图。桌面默认月视图，移动端默认列表视图，当前设备保留最近选择；提醒关闭但订阅有效时节点仍显示。
- 异常流程：没有订阅赛事时展示空状态并引导搜索赛事；同日节点过多时展开为紧凑列表；赛事节点更新后同步当前修订；赛事取消或紧急下架时不再展示未来节点；目标不可访问时显示状态而不提供失效跳转。
- 验收标准：三种视图读取同一节点事实并可无损切换；`primary` 节点、当前阶段和最近节点具有更强视觉层级，同阶段成对节点保留配对标签；同一天多个节点可访问；取消订阅后对应节点移除，关闭提醒但保留订阅时节点不移除；桌面和移动端均通过 Playwright 主路径。

### FR6 规则推荐与推荐理由

#### FR6.1 规则推荐列表

- 用户故事：作为本科学生，我希望系统根据我的画像推荐赛事，以便发现可能适合但尚未主动搜索的竞赛。
- 前置条件：系统存在已发布赛事；个性化模式需要一个独立审核激活的推荐规则集；用户画像可选。
- 主流程：仅当画像为 `recommendation_ready` 且存在有效激活规则集时，系统读取学生专业、年级和兴趣标签，按受控规则匹配赛事适合专业、适合年级、类别、参考标签和截止时间并生成个性化理由；目标偏好可作为附加信号，但不决定画像完成状态。响应标识所用规则集版本。
- 异常流程：用户未登录、画像为 `incomplete` 或没有有效激活规则集时，明确降级到通用 `actionable` 推荐并返回降级依据；不得使用隐藏 service 常量伪装个性化。没有足够匹配赛事时补充近期可报名赛事；赛事已下架或取消时从推荐中移除；缺少激活规则集同时向管理端暴露配置异常。
- 验收标准：推荐列表只展示已发布且未下架赛事；每条推荐必须展示至少一个推荐理由或通用排序依据；响应明确本次使用个性化还是通用模式及规则集版本；推荐结果不得依赖机器学习模型、可执行规则脚本或代码常量作为当前个性化实现前提。

#### FR6.2 推荐理由展示

- 用户故事：作为本科学生，我希望看到推荐原因，以便判断推荐是否可信。
- 前置条件：推荐结果已生成；推荐规则返回匹配原因。
- 主流程：系统在推荐条目中展示理由，例如“与你的专业匹配”“适合当前年级”“与你关注的人工智能方向相关”“报名截止较近”。
- 异常流程：没有个性化匹配原因时展示通用原因，例如“近期发布”或“近期截止”；推荐依据缺失时不展示空标签。
- 验收标准：推荐理由必须来自明确规则；同一条赛事最多展示 3 个主要理由；推荐理由与详情页适配标签不应互相矛盾。
- 规则治理：推荐规则集按版本保存，草稿可编辑，提交后快照不可改写；另一名审核者可通过、驳回或退回，激活时原子替换当前版本并退役旧版本。提交者不得自审，内部权重和排序分数不得公开。

#### FR6.3 推荐曝光与点击统计

- 用户故事：作为产品治理管理员，我希望看到推荐结果的记录曝光与点击，以便判断入口是否被使用，而不追踪具体学生或把点击误当成报名。
- 前置条件：推荐 API 已返回带一次性 request ID、位置、模式、规则集版本和受控理由代码的结果。
- 主流程：前端实际渲染推荐项后批量尽力记录曝光；学生从推荐结果进入赛事详情时尽力记录点击。服务端只接受该 request 中真实返回的赛事，并从服务端快照读取位置、模式、规则版本和理由。
- 异常流程：统计请求失败、超时或被浏览器拦截时不影响推荐展示或详情导航；重复曝光或点击请求幂等；伪造或过期 request/item 被拒绝；未实际渲染的 API 返回项不计曝光。
- 验收标准：原始记录只包含随机 request ID、赛事届次、曝光/点击时间、位置、推荐模式、规则集版本、受控理由代码和登录状态类别，不包含用户 ID、账号标识、画像字段、IP、User-Agent 或跨 request/跨日访客标识；P2 不用这些事件自动调整个人推荐。
- 数据保留：原始 request/item 记录保留 90 天，随后分别保留 item-level 总量和 reason-level 归因。曝光按 `impressed_at` 的上海日期、点击按 `clicked_at` 的上海日期进入聚合，同一 request item 可跨日贡献两类事件；总量还按规则集版本、模式、位置、登录状态类别和赛事区分，每个 request item 对每类事件最多计一次。理由归因在相同维度上增加理由代码，一个多理由 item 可进入多个理由行，因此理由行不得相加为总体。管理端 7/30 日比值是窗口内点击事件除以窗口内曝光事件的 best-effort 互动比值，不是按曝光日 cohort 计算的转化率；理由行标注为理由归因，二者均不代表独立用户、因果效果、推荐质量或报名转化。

### FR7 增强内容能力

以下能力保留为增强范围，当前先明确稳定的功能边界和扩展方向；进入具体排期后，可在不改变业务对象和权限边界的前提下补充任务级验收标准。

- 资料归档：管理员或授权用户可上传往届通知、优秀作品、经验材料、常见问题和复盘资料，并关联到赛事或类别。
- 组队交流：学生可围绕指定赛事发布组队需求，说明技能要求、人数、联系方式和截止时间。
- 认证答疑：参加过竞赛或获得奖项的用户可申请认证，认证通过后在答疑内容中展示身份标签。
- 内容审核：论坛、资料、认证申请和用户反馈均需进入审核流程，审核通过后才能公开展示。

### FR8 后台运营与统计

后台运营与统计用于支撑内容质量和后续优化，当前需求保留主要能力边界：

- 用户与权限管理：只有具备 `user_administrator` 的管理员可查看治理用户列表，并修改其他账号的角色、状态和受控 capability 集合；不得修改自身，也不得禁用、降级或撤销最后一个有效用户治理管理员。每次成功变更都使目标账号现有 session 失效，并记录原因及字段白名单内的新旧值。
- 基础配置管理：管理员可维护赛事类别、参考标签、适合专业、适合年级和站内消息模板；推荐权重属于独立版本化推荐规则集，通过编辑、合成预览、差异审核和激活流程维护，不允许直接原地修改线上规则。
- 内容审核：管理员可审核赛事、资料、认证申请和用户反馈。
- 统计分析：系统可统计访问量、搜索量、收藏量、订阅量、官方通道跳转量及推荐曝光/点击。外链统计仅代表站内记录的点击次数；推荐比值仅代表同一事件窗口内已记录点击事件数/曝光事件数，不是 cohort 转化率，二者均不代表独立用户、推荐质量或报名转化。
- 治理工作台：P2 thin 必须同时提供审核、审计和统计三个只读证据标签页。审核记录支持按目标类型、状态、提交者和时间筛选并查看差异、影响和意见；审计支持按操作者、动作、目标和日期筛选；统计展示当前内容/待办状态和 7/30 日关键互动计数，每项带口径、数据截至时间和 best-effort 限制。
- 首页摘要：治理首页只展示待办数量、推荐配置异常和少量关键摘要，详细信息进入对应标签页；P2 不建设实时大屏、复杂 BI、导出中心或用户级钻取。
- 证据约束：审核决定和审计日志创建后不可通过治理界面修改；统计只读。密码、验证码、session、完整账号标识、用户画像内容和其他敏感值不得写入审核/审计 detail 或统计维度。

## 七、非功能需求

### NFR1 性能

- 赛事列表、搜索和筛选在常规数据量下响应时间应不超过 3 秒。
- 赛事详情页在常规网络环境下首屏内容加载时间应不超过 3 秒。
- 推荐列表加载失败时不能阻塞基础搜索和赛事列表访问。

### NFR2 数据时效性

- 管理员审核通过的赛事应在前台立即或近实时可见。
- 赛事状态变更后，列表、详情、推荐和订阅提醒应同步更新。
- 已过报名截止时间的赛事应自动或通过后台任务标记报名状态，避免继续展示为可报名。
- 学生端赛事日期展示和日期筛选统一使用 `Asia/Shanghai` 产品日历时区。

### NFR3 易用性

- 学生完成一次搜索、筛选、进入详情、收藏或订阅的路径应清晰可见。
- 无结果、未登录、无订阅、链接不可用等状态必须提供明确提示和下一步入口。
- 赛事列表和详情页应优先展示学生决策所需字段，包括截止时间、适配标签、参赛形式和官方链接。
- 赛事治理工作台采用紧凑、可扫描的工作界面，编辑、校验、差异审核和状态维护有明确层级；P1 完成标准包含两个独立管理员从编辑提交到审核发布的 Playwright 主路径证据。

### NFR4 安全与权限

- 学号、手机号、邮箱和画像信息应按用户隐私数据处理，普通用户不能访问他人画像。
- 注册验证码必须随机、单次、限时、限制尝试次数并以不可恢复形式存储；生产日志和 API 响应不得包含验证码。
- 密码必须在 NFC 规范化后完整交给显式配置参数的自适应算法哈希，不得明文存储、可逆加密、静默截断或依赖框架默认哈希参数。优先使用 Argon2id；不可用时才使用达到当前安全基线的 scrypt。
- 登录失败响应不得泄露账号存在、待激活或被禁用状态，并应按账号标识和请求来源渐进限速；不得用可被攻击者触发的永久锁定代替限速。
- 每个受保护请求必须重新校验账号为 `active` 且 Cookie 中会话版本与账号一致；账号禁用、确认凭据泄露或终止全部会话时递增版本，使既有 session 在下一请求失效。
- 学生 session 空闲 24 小时或签发满 7 天后失效；管理员 session 空闲 30 分钟或签发满 8 小时后失效。超时由服务端校验，P1 不提供用户可选的“记住我”。
- 后台管理、赛事审核、状态变更、配置修改和内容处理必须进行身份鉴权。
- 关键后台操作必须记录操作人、时间、对象、操作类型和结果。

### NFR5 可靠性

- 站内提醒生成应避免重复提醒和漏提醒。
- 赛事节点时间修改后，未发送提醒必须重新计算。
- 官方链接不可用时，系统应给出明确提示，但不应删除原始来源记录。

### NFR6 兼容性

- 学生端应兼容主流桌面浏览器。
- 核心查赛、详情、收藏订阅和消息查看流程应适配移动端浏览。
- 后台管理优先面向桌面端，但关键表单和审核信息不得依赖过窄屏幕才能完整查看。

### NFR7 可维护性与扩展性

- 赛事类别、参考标签、适合专业、适合年级、推荐规则权重、消息模板应可配置。
- 人工录入流程应为后续半自动采集预留候选来源、重复提示和审核入口。
- 推荐规则应保持可解释，避免将不可追溯的算法结果直接作为赛事价值判断依据。

## 八、约束条件

- 账号唯一约束：每个学号、手机号或邮箱只能绑定一个有效账号。
- 密码策略约束：P1 不要求大小写、数字或符号组合，不做周期性强制换密；仅在确认泄露或安全处置时要求更换。密码阻止列表在本地执行，不依赖在线泄露查询服务。
- 注册渠道约束：类型化标识模型不代表所有标识都开放自助注册。P1 的邮箱注册由可用发送方配置控制；手机号短信注册延后；无投递渠道的部署关闭公开注册并使用受控账号配置路径。
- 机构边界约束：当前部署只服务一个配置确定的高校；未来多高校共享必须重新设计机构实体、租户级唯一性、权限隔离和配置治理，不以新增学校字符串代替。
- 来源可信约束：赛事信息应优先来自学校官网、学院公告、竞赛官网、官方通知等可信来源，并保留来源标识。
- 价值说明约束：适配标签和价值依据仅用于辅助学生选赛，不替代学校、学院或官方对竞赛级别、加分政策和评奖规则的正式认定。
- 提醒渠道约束：当前版本仅定义站内提醒，不包含邮件、短信、微信或企业微信等外部提醒；可选的账号验证邮件只用于身份激活，不构成赛事提醒渠道。
- 权限边界约束：普通学生不能访问后台管理功能；竞赛组织者不能直接修改系统推荐规则、参考标签和权限配置。
- 外部链接约束：报名链接和附件下载依赖外部官方站点，系统需提示外部站点异常风险，并保留官方来源入口。
- 日志留痕约束：审核、配置修改、权限调整、内容处理、赛事状态变更必须记录日志。

## 九、默认设计决策

- 交付方式：PRD 保持功能需求完整描述，后续实现按核心能力、增强能力和后续能力分层推进，不将 PRD 本身作为严格验收清单。
- 教师与教学秘书：当前不单独设计独立工作台，优先复用后台统计、赛事管理和内容审核能力；后续如出现明确高频工作流，再拆分专属视图。
- 学生竞赛经历：默认由学生自填，用于画像、推荐和展示；仅当经历被用于认证答疑、官方荣誉或权限授予时，才进入材料提交和管理员审核流程。
- 赛事参考标签：当前由系统管理员统一配置和维护，保证标签口径稳定；后续可扩展为学院级标签，但学院标签不得覆盖系统级基础标签和来源事实。
- 资料归档、组队交流和认证答疑：作为增强或后续能力保留，设计上必须复用账号、赛事、审核和权限体系，避免形成独立内容孤岛。
- 扩展性要求：后续扩展应优先通过配置、状态扩展和模块化页面增加能力，避免破坏既有核心链路的数据结构和用户路径。
