# 1．引言

## 1.1 编写目的

本文档用于说明“大学生竞赛信息智能筛选与推荐系统”的开发目标、工作内容、阶段计划、人员分工、交付物和验收标准。文档面向课程指导教师、项目组成员、后续开发维护人员和答辩评审人员，作为项目组织、进度安排和阶段验收的依据。

本文档同时区分课程报告交付与工程实现路线：`01_项目开发计划.md`、`02_需求规格说明.md`、`03_软件设计说明.md` 是当前需要立即完成的课程交付材料；系统实现按照 `docs/roadmap.md` 中的工程阶段逐步推进。

## 1.2 项目背景

本项目来源于软件工程课程设计。项目正式名称为“大学生竞赛信息智能筛选与推荐系统”，英文短名为 CompeteHub。

| 项目项 | 内容 |
|---|---|
| 委托单位 | 软件工程课程设计课程组 |
| 开发单位 | 第七组 |
| 主管部门 | 学院或课程教学管理单位 |
| 服务对象 | 在校本科生、系统管理员，以及教师、教学秘书、竞赛组织者等辅助干系人 |

在校本科生获取竞赛信息时常面临信息渠道分散、筛选成本高、适配程度难判断、关键时间节点容易遗漏等问题。本系统面向学生发现、判断和跟进赛事的完整任务链路，提供可信赛事信息维护、搜索筛选、详情展示、收藏订阅、站内提醒、个人赛事日历和规则推荐等能力。

系统与其他系统的关系如下：

- 前台学生端通过浏览器访问系统，完成赛事发现、订阅和提醒查看。
- 后台管理端由管理员维护赛事、审核发布、配置基础规则和查看审计统计。
- 赛事来源来自学校官网、学院公告、竞赛官网、官方通知等可信来源，系统保留来源标识和官方链接。
- 当前版本不直接接入邮件、短信、微信或第三方报名系统，外部链接仅作为官方通道跳转。

# 2．项目概述

## 2.1 工作内容

项目工作内容分为课程文档交付和工程实现两类。

课程文档交付包括：

- 完成项目开发计划、需求规格说明书和软件设计说明书。
- 对齐术语、范围、模块划分、数据对象、接口和系统结构。
- 使用 Markdown 表格、Mermaid 图和必要的 UML 表达系统设计。

工程实现工作包括：

- 搭建可运行的 Vue 前端、Flask 后端、PostgreSQL、Redis 和 Celery 基础环境。
- 实现学生注册登录、画像维护、赛事搜索筛选、详情查看、收藏订阅、站内提醒和个人赛事日历。
- 实现管理员赛事人工录入、审核发布、状态维护、配置管理、审计日志和基础统计。
- 实现规则推荐、推荐排序、推荐理由和解释一致性校验。
- 为后续内容沉淀、组队交流、认证答疑、半自动采集候选和外部提醒预留扩展空间。

## 2.2 条件与限制

已具备条件：

- 已确定项目名称、核心术语、当前版本范围和七模块划分。
- 仓库已有 Vue 前端、Flask 后端、PostgreSQL/Redis 本地基础设施和 MkDocs 文档站。
- 已有 PRD、架构、API、数据模型、技术规格和模块拆分文档可作为报告素材。

尚需创造或完善的条件：

- 完成 01/02/03 正式课程报告。
- 补齐后端 API、服务层、数据库迁移、测试数据和业务测试。
- 补齐前端页面、交互状态、接口联调和演示数据。
- 在实现推进中持续同步需求、架构、API、数据模型和模块文档。

主要限制：

- 当前版本采用管理员人工录入赛事，不承诺自动采集和自动发布。
- 当前版本只承诺站内提醒，不承诺邮件、短信、微信或小程序推送。
- 当前版本不公开赛事价值评分，不使用“含金量分数”作为功能承诺。
- 教师、教学秘书和竞赛组织者作为辅助干系人或预留角色，不提供独立工作台。
- M7 内容沉淀与交流扩展不作为当前验收主线。

## 2.6 验收标准

课程报告验收标准：

- 01/02/03 三份正式报告结构完整，术语统一，范围清楚。
- 报告不再承诺公开赛事评分、自动采集发布、外部提醒和独立教师/组织者工作台。
- 报告中的模块、数据、接口和流程与当前七模块口径一致。
- Markdown、Mermaid 图和表格可通过 `just docs-build` 构建。

工程阶段验收标准：

- 本地可通过文档命令启动前端、后端、PostgreSQL 和 Redis。
- 管理员能够创建赛事、提交审核、审核发布和维护状态。
- 学生能够搜索筛选赛事、查看详情、收藏订阅、查看站内提醒和个人赛事日历。
- 推荐结果能够展示可追溯的推荐理由，且不公开绝对价值评分。
- 关键后台操作具备审计记录，权限由后端进行控制。

## 3.3 预算

本项目以课程设计和本地开发为主，不涉及商业采购预算。主要成本为成员时间、开发设备和必要的开源工具使用。

| 项目 | 预算说明 |
|---|---|
| 开发工具 | 使用开源工具和免费开发环境，无额外软件授权费用。 |
| 服务器 | 当前阶段使用本地 Docker Compose，不采购云服务器。 |
| 第三方服务 | 当前版本不接入短信、邮件、微信等付费通知服务。 |
| 人力成本 | 由第七组成员按课程任务投入。 |
| 风险预留 | 若后续需要部署演示环境，可再评估云主机或数据库托管费用。 |

## 3.4 关键问题

| 关键问题 | 风险 | 对策 |
|---|---|---|
| 范围膨胀 | 自动采集、外部提醒、公开评分等能力会扩大实现成本。 | 当前版本只承诺人工录入、站内提醒和规则推荐；其他能力列为后续扩展。 |
| 文档漂移 | PRD、架构、模块和报告可能出现口径不一致。 | 以 `CONTEXT.md` 和 `docs/roadmap.md` 为对齐入口，相关文档一起精修。 |
| 提醒可靠性 | 节点变化可能导致重复提醒或漏提醒。 | 以 PostgreSQL 的提醒记录为事实来源，后台任务保持幂等。 |
| 权限边界 | 前端隐藏按钮不能替代安全控制。 | 后端接口强制鉴权和角色校验，关键操作写审计日志。 |
| 推荐可信度 | 推荐理由若不可解释会降低信任。 | 使用规则推荐，不公开赛事价值分数，推荐理由来自明确规则或赛事字段。 |
| 中文搜索质量 | 简单数据库匹配可能无法覆盖复杂中文检索。 | 初期采用 PostgreSQL 过滤和关键词匹配，后续视需要引入专用搜索服务。 |
