## 一、项目概述

“大学生竞赛信息智能筛选与推荐系统”面向在校本科生、辅导员/教学秘书、学生社团与竞赛组委会等对象，旨在解决竞赛信息分散、筛选成本高、赛事价值难判断、关键时间节点容易遗漏等问题。系统希望通过信息聚合、智能筛选、价值评估、订阅提醒和个性化推荐等手段，帮助用户更高效地发现适合自己的竞赛。

系统的核心业务目标可以概括为四点：第一，统一收集并管理高校竞赛信息；第二，帮助学生快速筛选与自身专业、年级、能力相匹配的赛事；第三，为用户提供报名、截稿、答辩等关键节点提醒；第四，沉淀历届参赛数据与经验材料，形成长期服务闭环。

## 二、需求获取方法说明

本报告采用“问卷法 + 访谈法 + 业务流程分析法”三种方式获取初步需求。其中，问卷法主要用于大范围了解学生群体的普遍痛点；访谈法主要用于挖掘不同角色的深层需求；业务流程分析法主要用于梳理系统管理员和平台运营侧的管理需求。

| 方法      | 适用对象               | 目的                | 形成记录        |
| ------- | ------------------ | ----------------- | ----------- |
| 问卷法     | 在校本科生              | 获取高频痛点、使用习惯、功能偏好  | 问卷题目摘要、结果汇总 |
| 访谈法     | 学生、辅导员/教学秘书、社团/组委会 | 获取角色差异化需求与业务细节    | 访谈纪要、需求摘录   |
| 业务流程分析法 | 系统管理员/运营人员         | 明确后台审核、维护、配置与统计需求 | 流程分析记录、约束整理 |

## 四、需求获取记录

### 4.1 面向在校本科生的需求获取记录

在校本科生是系统最核心的用户群体，因此采用“问卷法 + 半结构化访谈”并行获取需求。问卷用于获得普遍性结论，访谈用于深入理解具体使用场景。

#### 4.1.1 问卷记录摘要

问卷目标：了解学生在寻找竞赛信息、筛选竞赛、评估竞赛价值、追踪时间节点等方面的真实痛点。问卷对象主要为本科生，覆盖大一至大四多个年级和不同专业方向。根据前期材料，团队已回收有效问卷 112 份，其中 86% 的受访学生明确表示“获取竞赛信息存在困难，曾因信息差错失过参赛机会”。

| 题号 | 问卷核心问题                 | 记录的主要结论                       |
| -- | ---------------------- | ----------------------------- |
| Q1 | 你通常通过什么渠道获取竞赛信息？       | 官网、学院群、公众号、同学转发较为常见，渠道分散明显。   |
| Q2 | 你是否曾因信息不及时而错过比赛？       | 大部分受访者表示有类似经历，信息时效是关键问题。      |
| Q3 | 你最希望系统优先提供什么能力？        | 高频答案集中在“集中检索”“精准筛选”“报名截止提醒”。  |
| Q4 | 你是否关心比赛含金量、保研加分、简历认可度？ | 绝大多数学生表示非常关心，但现有渠道难以判断。       |
| Q5 | 你是否愿意接受个性化推荐？          | 多数学生愿意，但要求推荐理由清楚、可手动关闭。       |
| Q6 | 你希望查看哪些历史信息？           | 往届获奖作品、赛事难度、指导老师经验、比赛流程被多次提及。 |

由问卷可以提炼出学生群体的共性需求：一是需要一个统一入口集中查看竞赛信息；二是希望通过专业、年级、赛事级别、截止日期等条件快速筛选；三是希望系统能提供赛事价值评估和历史经验参考；四是希望系统主动提醒关键节点，减少遗忘或错过。

#### 4.1.2 访谈记录摘要

| 访谈对象 | 角色背景         | 典型表述                           | 提炼需求                  |
| ---- | ------------ | ------------------------------ | --------------------- |
| 学生A  | 大二，计算机类，比赛小白 | “我知道比赛很多，但不知道哪些适合我，怕白花时间。”     | 需要新手友好的入门筛选与推荐功能。     |
| 学生B  | 大三，准备保研      | “我更关心竞赛是否被学院认可、含金量高不高、是否值得投入。” | 需要赛事价值评估、认可度信息与优先级排序。 |
| 学生C  | 大四，求职导向      | “我经常在群里刷到消息，但链接太散，截止日期也容易漏。”   | 需要官方链接聚合、收藏订阅、时间提醒。   |

### 4.2 面向辅导员/教学秘书的需求获取记录

辅导员与教学秘书虽然不是系统的最大流量用户，但属于高影响力角色。他们既需要向学生发布竞赛信息，也需要统计参与情况，因此采用访谈法进行需求获取。

| 记录项  | 内容                                             |
| ---- | ---------------------------------------------- |
| 访谈对象 | 辅导员/教学秘书（匿名）                                   |
| 主要痛点 | 竞赛通知通常依赖班群或年级群逐条转发，重复劳动较多；后期统计学生报名与获奖情况时工作量较大。 |
| 重点表述 | “如果系统能把适合本学院学生的比赛自动筛出来，并且支持一键推送、导出统计表，就能省很多事。” |
| 提炼需求 | 需要学院视角的赛事筛选、批量通知、学生报名/参与统计、导出报表与历史记录查询功能。      |

### 4.3 面向学生社团/竞赛组委会的需求获取记录

学生社团或竞赛组委会是系统中重要的信息供给方与宣传方，适合采用访谈法获取需求。

| 记录项  | 内容                                             |
| ---- | ---------------------------------------------- |
| 访谈对象 | 竞赛宣发负责人/学生社团骨干（匿名）                             |
| 主要痛点 | 传统宣传方式覆盖面有限，难以触达真正匹配专业和年级的学生；活动链接、附件和时间节点维护分散。 |
| 重点表述 | “我们更希望把比赛推给真正可能报名的人，而不是简单在群里刷屏。”               |
| 提炼需求 | 需要赛事发布、目标人群标签设置、官方链接/附件维护、宣发效果统计、报名热度反馈等能力。    |

### 4.4 面向系统管理员的业务流程分析记录

系统管理员虽然不一定直接参与竞赛，但必须保证数据准确性、推荐规则合理性和平台稳定运行，因此采用业务流程分析法进行需求获取。

| 流程环节  | 现状问题            | 管理员侧需求                |
| ----- | --------------- | --------------------- |
| 信息采集  | 来源分散、格式不统一、可能重复 | 需要支持采集、去重、人工复核与来源追踪。  |
| 内容维护  | 过期赛事若不及时处理会误导学生 | 需要对赛事状态进行上架、下架、归档管理。  |
| 评分配置  | 不同赛事价值评价标准不一致   | 需要可配置的评分规则与解释说明。      |
| 用户与权限 | 不同角色可操作范围不同     | 需要角色权限控制、日志记录与异常处理。   |
| 统计分析  | 难以衡量推荐效果与使用效果   | 需要访问统计、收藏订阅统计、报名转化统计。 |

### 5.2 功能需求详细描述

#### M1 用户与画像管理模块

##### FR1.1 注册与登录功能

内容：用户填写必要信息后可注册成为系统用户，并通过账号密码或验证码登录系统。

输入：学号/手机号/邮箱、密码、验证码、角色选择信息。

输出：注册成功后进入系统首页；若失败则返回错误原因或校验提示。

##### FR1.2 个人画像维护功能

内容：学生可维护专业、年级、兴趣方向、竞赛经历、目标导向（保研/求职/兴趣提升）等个人画像信息。

输入：专业、年级、兴趣标签、竞赛经历、目标偏好。

输出：系统保存并更新用户画像，用于后续筛选与推荐。

##### FR1.3 收藏与订阅功能

内容：用户可对感兴趣的竞赛进行收藏或订阅，便于后续跟踪。

输入：竞赛ID、订阅方式、提醒偏好。

输出：系统生成收藏/订阅记录，并在个人中心展示。

#### M2 赛事信息采集与管理模块

##### FR2.1 赛事信息采集功能

内容：系统可从学校官网、学院通知、竞赛官网等渠道采集赛事信息并形成统一数据记录。

输入：来源网址、采集规则、原始赛事页面信息。

输出：生成标准化赛事记录并进入待处理列表。

##### FR2.2 赛事信息去重与清洗功能

内容：系统对重复赛事、格式不规范信息进行识别、合并和清洗，提升数据质量。

输入：原始赛事标题、时间、链接、主办方等字段。

输出：输出清洗后的结构化赛事数据，并标记重复关系。

##### FR2.3 赛事审核与状态管理功能

内容：管理员可对赛事进行审核、上架、下架、过期归档等管理。

输入：赛事记录、审核意见、状态变更操作。

输出：更新赛事状态，并在前台同步展示最新结果。

#### M3 搜索筛选与展示模块

##### FR3.1 关键词搜索功能

内容：用户可通过赛事名称、主办方、关键词等方式快速查询竞赛。

输入：关键词、搜索范围。

输出：返回符合条件的赛事列表，若无结果则提示用户调整条件。

##### FR3.2 多维筛选功能

内容：系统支持按专业、年级、赛事级别、报名状态、截止日期、团队/个人赛等条件进行筛选。

输入：筛选条件组合。

输出：输出按条件过滤后的赛事结果集。

##### FR3.3 排序与结果展示功能

内容：用户可按照截止时间、推荐度、赛事级别、热度等维度排序浏览结果。

输入：排序规则、分页参数。

输出：生成按规则排列的赛事列表和分页结果。

#### M4 赛事详情与价值评估模块

##### FR4.1 赛事详情展示功能

内容：系统展示赛事简介、报名条件、赛程安排、附件要求、官方链接等详情信息。

输入：赛事ID。

输出：输出完整赛事详情页。

##### FR4.2 赛事价值评估功能

内容：系统结合赛事级别、主办单位、往届关注度、学院认可情况等维度，对赛事进行参考性评分。

输入：赛事基础信息、评分规则、历史数据。

输出：输出赛事评分、等级说明和评分依据。

##### FR4.3 官方通道跳转功能

内容：用户在查看赛事详情后，可一键跳转至官方报名或通知页面。

输入：赛事ID、官方链接。

输出：打开对应官方页面并记录跳转行为。

#### M5 订阅提醒与日历模块

##### FR5.1 关键节点提醒功能

内容：系统可在报名截止、作品提交、答辩等关键时间节点前向订阅用户发送提醒。

输入：竞赛节点时间、用户提醒偏好、消息模板。

输出：生成站内消息或外部提醒通知。

##### FR5.2 个人赛事日历功能

内容：系统将用户收藏或订阅的赛事节点统一汇总到个人日历中。

输入：用户订阅记录、赛事时间节点。

输出：输出按时间排列的个人竞赛日历。

##### FR5.3 提醒设置管理功能

内容：用户可自定义是否提醒、提醒方式及提前天数。

输入：提醒方式、提前时间、消息开关。

输出：保存提醒配置，并按配置执行通知。

#### M6 个性化推荐模块

##### FR6.1 个性化竞赛推荐功能

内容：系统根据用户画像、浏览历史、收藏订阅行为向学生推荐更适合的竞赛。

输入：专业、年级、兴趣方向、行为日志等。

输出：输出个性化推荐列表。

##### FR6.2 推荐理由展示功能

内容：系统在推荐结果中给出简要推荐原因，如“与你的专业匹配”“距离截止时间较近”等。

输入：推荐结果、匹配规则。

输出：输出带解释信息的推荐条目。

##### FR6.3 推荐偏好调整功能

内容：用户可主动修改推荐偏好，降低不感兴趣类型的权重。

输入：偏好标签、屏蔽项、反馈信息。

输出：系统更新推荐模型输入并优化后续结果。

#### M7 资料沉淀与复盘模块

##### FR7.1 历届资料归档功能

内容：系统归档往届竞赛通知、优秀作品、经验分享与复盘资料。

输入：历史赛事资料、附件、标签信息。

输出：输出可按竞赛分类查询的资料库。

##### FR7.2 经验分享查看功能

内容：学生可查看参赛经验、备赛建议、常见问题等内容。

输入：竞赛类别、经验文章标签。

输出：输出相应经验内容。

##### FR7.3 赛后复盘记录功能

内容：参赛学生或指导教师可在赛后提交复盘记录，形成持续沉淀。

输入：复盘标题、正文、附件、关联赛事。

输出：系统保存复盘内容并关联到对应赛事。

#### M8 后台运营与统计模块

##### FR8.1 用户与权限管理功能

内容：管理员可管理不同角色的权限边界，保障系统安全运行。

输入：用户账号、角色信息、授权操作。

输出：更新权限结果并记录操作日志。

##### FR8.2 评分规则配置功能

内容：管理员可维护赛事评分相关的配置项与权重。

输入：评分因子、权重参数、适用范围。

输出：输出更新后的评分规则并应用于系统。

##### FR8.3 统计分析与反馈处理功能

内容：管理员可查看访问量、收藏量、订阅量、跳转量等统计信息，并处理用户反馈。

输入：行为日志、反馈内容、处理意见。

输出：生成统计报表并更新反馈处理状态。

### 5.3 非功能需求描述

| 编号   | 非功能需求   | 内容描述                                     | 涉及功能需求            |

| NFR1 | 性能需求    | 常规搜索、筛选与详情页加载时间应控制在 3 秒以内；高峰期系统仍应保持稳定响应。 | FR3.1~FR4.3       |
| NFR2 | 数据时效性需求 | 新抓取或新发布的赛事信息应尽快同步到前台；过期赛事应及时归档并标记状态。     | FR2.1~FR2.3       |
| NFR3 | 易用性需求   | 界面应清晰、操作流程简单，新用户应能在较短时间内完成搜索、筛选与订阅。      | FR1.1、FR3.1、FR5.1 |
| NFR4 | 安全性需求   | 系统应保护用户账号信息与行为数据，后台管理操作必须留痕，避免越权操作。      | FR1.1、FR8.1、FR8.3 |
| NFR5 | 可靠性需求   | 消息提醒、赛事状态更新、评分展示等关键服务应具备较高可靠性，避免误提醒与漏提醒。 | FR4.2、FR5.1、FR5.2 |
| NFR6 | 兼容性需求   | 系统应兼容主流桌面浏览器，并支持移动端访问或小程序/移动端适配。         | FR3.1~FR7.3       |
| NFR7 | 可维护性需求  | 赛事采集规则、评分规则、消息模板和角色权限应可配置，便于后续维护。        | FR2.1、FR8.1、FR8.2 |
| NFR8 | 可扩展性需求  | 后续应支持新增赛事来源、新增评分因子、新增推荐策略和新增用户角色。        | FR2.1、FR6.1、FR8.2 |

### 5.4 约束条件描述

| 编号 | 约束名称     | 内容                                      |
| -- | -------- | --------------------------------------- |
| C1 | 账号唯一     | 每个手机号、邮箱或学号只能绑定一个有效账号，避免重复注册和统计混乱。      |
| C2 | 数据来源约束   | 赛事信息应优先来自官方通知、学校官网、学院公告等可信来源，并保留来源标识。   |
| C3 | 评分使用约束   | 赛事价值评分仅用于辅助决策，不得替代学校或官方对竞赛级别、加分政策的正式认定。 |
| C4 | 提醒授权约束   | 站外提醒、消息推送等功能应在用户授权后才能启用，用户可随时关闭。        |
| C5 | 权限边界约束   | 不同角色只能在其授权范围内操作数据，例如组织者不能直接修改管理员配置项。    |
| C6 | 时效性约束    | 已截止、已取消或链接失效的赛事必须及时标记状态或下架，避免误导用户。      |
| C7 | 外部链接依赖约束 | 系统中的报名链接、附件下载等依赖外部官方站点，若对方站点异常，系统需给予提示。 |
| C8 | 日志留痕约束   | 涉及审核、配置修改、权限调整等后台操作必须记录日志，以便审计与追踪。      |

## 六、结论

通过对系统干系人的识别以及问卷、访谈、业务流程分析等多种方式的需求获取，可以得出结论：本系统最核心的需求来自学生群体，其次是辅导员/教学秘书与赛事组织方的信息发布和管理需求；系统的主要价值在于“聚合信息、降低筛选成本、提供价值判断、减少时间遗漏、沉淀长期资料”。

从初步需求整理结果看，系统至少应包含用户管理、赛事采集管理、搜索筛选、赛事详情与价值评估、订阅提醒、个性化推荐、资料沉淀、后台管理等八个功能模块，并同时满足性能、安全、时效性、可靠性和可维护性等非功能需求。
