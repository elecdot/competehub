# 三、系统技术栈选型

## 3.1 总体技术架构

本系统采用前后端分离架构，以 **Vue 3 + Flask + PostgreSQL + Redis** 为核心技术路线，面向大学生竞赛服务场景构建集竞赛信息管理、搜索筛选、个性化推荐、订阅提醒、交流论坛和后台运营于一体的 Web 系统。

系统整体可划分为前端展示层、后端接口层、数据存储层、缓存与异步任务层、数据采集处理层、文件存储层和部署运行层。各层之间通过 RESTful API 进行数据交互，使系统结构清晰、便于维护和扩展。

## 3.2 前端技术栈

前端采用 **Vue 3 + Vite + Vue Router + Pinia + Axios + Element Plus**。

| 技术 | 作用 |
|---|---|
| Vue 3 | 构建系统页面和交互逻辑 |
| Vite | 提供快速开发、构建和热更新能力 |
| Vue Router | 管理登录、赛事列表、赛事详情、推荐、论坛、后台等页面路由 |
| Pinia | 管理用户登录状态、用户信息、筛选条件、收藏订阅状态等全局数据 |
| Axios | 与 Flask 后端接口进行 HTTP 通信 |
| Element Plus | 快速构建表单、表格、分页、弹窗、菜单、后台管理界面等组件 |

前端主要页面包括：

- 登录注册页面
- 用户中心页面
- 赛事列表与搜索筛选页面
- 赛事详情与价值评估页面
- 个性化推荐页面
- 订阅日历与提醒设置页面
- 交流论坛与资料库页面
- 后台管理页面

采用该前端技术组合的原因是 Vue 3 学习成本适中，生态成熟，适合课程设计项目快速开发；Element Plus 能提升后台管理、表格和表单类页面的开发效率。

## 3.3 后端技术栈

后端采用 **Python 3.x + Flask + Flask Blueprint + Flask-JWT-Extended + SQLAlchemy + Flask-Migrate**。

| 技术 | 作用 |
|---|---|
| Python 3.x | 实现业务逻辑、数据处理、推荐算法和统计分析 |
| Flask | 提供后端 Web 服务和 RESTful API |
| Flask Blueprint | 按业务模块拆分后端代码 |
| Flask-JWT-Extended | 实现基于 JWT 的登录认证和接口鉴权 |
| SQLAlchemy | 实现数据库对象关系映射，简化数据操作 |
| Flask-Migrate | 管理数据库表结构迁移 |
| Marshmallow / Pydantic | 可选，用于接口参数校验和数据序列化 |

后端建议按以下 Blueprint 组织：

- `auth`：注册、登录、身份认证
- `user`：用户中心与画像管理
- `competition`：赛事信息、搜索筛选、详情展示
- `recommend`：个性化推荐与推荐理由
- `reminder`：收藏订阅、节点提醒、个人日历
- `forum`：交流论坛、组队、认证答疑、资料沉淀
- `admin`：后台管理、审核、规则配置、统计分析

该后端方案能够较好支撑系统的八大功能模块，同时保持代码结构清晰。

## 3.4 数据库与缓存技术栈

系统采用 **PostgreSQL** 作为主数据库，采用 **Redis** 作为缓存和异步任务中间件。

### PostgreSQL

PostgreSQL 用于存储系统核心业务数据，包括：

- 用户信息与角色信息
- 用户画像、兴趣标签、竞赛经历
- 赛事基础信息、赛事节点、官方链接、附件信息
- 收藏、订阅、提醒配置
- 浏览、点击、跳转等行为日志
- 推荐记录与推荐理由
- 论坛帖子、评论、组队信息、认证申请
- 历届资料、经验分享、赛后复盘
- 后台审核记录、操作日志、统计数据

选择 PostgreSQL 的原因是其支持复杂查询、事务管理和较强的数据一致性，适合竞赛信息筛选、后台统计和多表关联场景。

### Redis

Redis 主要用于：

- 缓存热门赛事列表
- 缓存推荐结果
- 存储验证码或临时登录状态
- 缓存高频查询结果
- 作为 Celery 的 Broker

Redis 能减少数据库压力，提升系统高频访问接口的响应速度。

## 3.5 数据采集与处理技术栈

赛事信息来源分散，系统需要具备基础的数据采集与清洗能力。数据采集与处理采用 **Requests + BeautifulSoup4 + lxml + pandas + python-dateutil**。

| 技术 | 作用 |
|---|---|
| Requests | 获取学校官网、学院通知或竞赛官网页面内容 |
| BeautifulSoup4 | 解析 HTML 页面中的标题、时间、主办方、链接等信息 |
| lxml | 提升 HTML/XML 解析效率 |
| pandas | 对采集结果进行清洗、去重和标准化处理 |
| python-dateutil | 统一处理报名时间、截止时间、比赛时间等日期字段 |

如果课程设计阶段不实现完整自动采集，也可以先采用“后台人工录入 + Excel/CSV 辅助导入”的方式落地，但技术选型中保留采集处理能力，有利于体现系统扩展性。

## 3.6 推荐算法与数据分析技术栈

个性化推荐模块采用 **规则推荐 + 标签加权匹配** 作为第一阶段实现方案，后续可扩展为基于相似度或协同过滤的推荐方式。

推荐相关技术包括：

- NumPy
- pandas
- scikit-learn
- 自定义规则推荐算法

第一阶段推荐依据包括：

- 用户专业
- 用户年级
- 兴趣标签
- 浏览历史
- 收藏记录
- 订阅记录
- 赛事类别
- 赛事级别
- 截止时间
- 赛事热度

推荐分值可由专业匹配度、兴趣匹配度、历史行为相似度、时间紧迫度和热门程度等因素组成。系统在展示推荐结果时，应同时展示推荐理由，例如“与你的专业匹配”“你曾收藏过相似竞赛”“该比赛距离截止时间较近”。

采用规则推荐的原因是其实现难度适中、可解释性强，适合课程设计阶段落地。

## 3.7 异步任务与定时调度技术栈

系统采用 **Celery + Redis + Celery Beat / APScheduler** 处理异步任务和定时任务。

主要应用场景包括：

- 定时采集赛事信息
- 定时检查即将截止的赛事
- 定时发送订阅提醒
- 定时刷新推荐结果
- 批量生成统计数据
- 处理大批量数据清洗任务

课程设计阶段可优先实现基础定时任务，例如每日检查即将截止赛事和更新赛事状态；完整 Celery 任务队列可作为增强功能。

## 3.8 文件存储技术栈

系统涉及头像、赛事附件、论坛附件、认证证明材料、历届作品和复盘资料等文件，因此需要文件存储能力。

第一阶段采用 **本地文件存储**，实现简单，便于开发和演示。后续可扩展为 **MinIO** 对象存储，用于统一管理文件访问、权限和容量扩展。

## 3.9 安全与权限控制技术栈

系统安全方案采用 **JWT + RBAC + 密码加密存储 + 日志审计**。

| 技术或机制 | 作用 |
|---|---|
| JWT | 实现登录认证和接口访问鉴权 |
| RBAC | 按普通用户、认证用户、教师或管理员等角色控制权限 |
| 密码加密存储 | 避免用户密码明文保存 |
| 日志审计 | 记录后台审核、权限调整、规则配置等敏感操作 |
| 验证码 / 限流 | 可选，用于减少恶意注册、暴力登录和高频请求 |

该方案能够满足用户数据保护、后台权限边界和操作追踪等安全需求。

## 3.10 测试技术栈

系统测试阶段采用以下工具：

- **pytest**：用于后端单元测试和推荐规则测试。
- **Postman / Apifox**：用于接口测试，包括登录、赛事查询、推荐、论坛和后台管理接口。
- **浏览器开发者工具**：用于前端页面调试和接口联调。

重点测试内容包括：

- 注册登录与权限控制是否正确
- 搜索筛选结果是否准确
- 赛事详情和状态展示是否正确
- 收藏订阅和提醒配置是否生效
- 推荐结果和推荐理由是否符合规则
- 后台审核和统计功能是否可用

## 3.11 部署与运行环境技术栈

部署层采用 **Nginx + Gunicorn + Linux 服务器 / 虚拟机**，Docker 作为可选增强方案。

| 技术 | 作用 |
|---|---|
| Nginx | 托管前端静态资源，进行反向代理和请求转发 |
| Gunicorn | 运行 Flask 后端服务 |
| Linux 服务器 / 虚拟机 | 作为系统部署运行环境 |
| Docker | 可选，用于统一部署 PostgreSQL、Redis、Flask 等服务 |

课程设计演示阶段可使用本地开发环境运行前后端；正式部署说明中可采用 Nginx 和 Gunicorn 作为推荐方案。

## 3.12 技术栈与功能模块对应关系

| 功能模块 | 对应技术栈 | 主要用途 |
|---|---|---|
| M1 用户与画像管理模块 | Vue 3、Flask、JWT、SQLAlchemy、PostgreSQL | 实现注册登录、个人画像、收藏订阅和权限控制 |
| M2 赛事信息采集与管理模块 | Flask、Requests、BeautifulSoup4、pandas、PostgreSQL | 实现赛事采集、清洗、去重、审核和状态管理 |
| M3 搜索筛选与展示模块 | Vue 3、Axios、Flask、PostgreSQL、Redis | 实现关键词搜索、多维筛选、排序和分页展示 |
| M4 赛事详情与价值评估模块 | Vue 3、Flask、PostgreSQL、规则评分算法 | 实现赛事详情、价值评分、适配人群说明和官方链接跳转 |
| M5 订阅提醒与日历模块 | Vue 3、Flask、PostgreSQL、Redis、Celery | 实现收藏订阅、节点提醒、个人日历和提醒设置 |
| M6 个性化推荐模块 | Flask、NumPy、pandas、scikit-learn、Redis | 实现用户画像分析、规则推荐、标签匹配和推荐结果缓存 |
| M7 交流论坛、资料沉淀与复盘模块 | Vue 3、Flask、JWT、PostgreSQL、本地文件存储 | 实现发帖评论、组队交流、认证答疑、资料归档和复盘记录 |
| M8 后台运营与统计模块 | Vue 3、Element Plus、Flask、PostgreSQL、Redis、Celery | 实现用户权限、内容审核、规则配置、统计分析和日志审计 |

## 3.13 技术选型总结

综上，本系统最终采用以 **Vue 3 + Flask + PostgreSQL + Redis** 为核心的前后端分离架构。前端通过 Vue 3 和 Element Plus 实现主要页面与后台管理界面；后端通过 Flask、Blueprint、JWT 和 SQLAlchemy 实现接口服务、权限控制和数据访问；数据库采用 PostgreSQL 存储核心业务数据；Redis 用于缓存和异步任务支撑；Celery 用于定时采集、提醒和统计任务；推荐模块第一阶段采用规则推荐与标签加权匹配方案。

该技术路线能够支撑用户与画像管理、赛事采集管理、搜索筛选、赛事详情与价值评估、订阅提醒、个性化推荐、交流论坛与资料沉淀、后台运营统计八大模块，具有较好的可实现性、可维护性和扩展性，适合作为软件工程课程设计项目的技术方案。
